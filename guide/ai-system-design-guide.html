Interview Format Guide

AI-Assisted
System Design

A new interview format where your depth drives the AI — and the quality of your prompts signals your engineering seniority.

What is this format?

Traditional system design expects you to narrate and whiteboard in real time. The AI-assisted variant changes the medium: you design alongside an AI tool — using it to generate artifacts like API specs, DDL schemas, sequence diagrams, and failure tables — while you supply the architectural judgment that drives those artifacts.

The interviewer evaluates two axes simultaneously:

📐 System Design Depth

Do you understand trade-offs? Can you make the right choices about architecture, data stores, consistency models, and scale — without relying on AI to decide for you?

🤖 AI Collaboration Skill

Do your prompts reflect domain expertise? Do you course-correct when AI goes wrong? Do you use AI to accelerate rather than substitute for your thinking?

Mental Model

Think of AI as a fast, capable junior engineer who is great at generating artifacts but lacks architectural judgment. You are the Staff Engineer who scopes the problem, makes the architectural calls, names the components, and reviews what the junior produces. A great Staff Engineer doesn't say "design me a Twitter" — they say "given our 500M DAU and 10:1 read/write ratio, compare fan-out-on-write vs fan-out-on-read for the home timeline service, specifically for accounts with 50M+ followers."

Three AI Interaction Modes

🏗️ Generate

You define the decision; AI produces the artifact. "I'm using Base62 + Cassandra. Generate the DDL and the redirect API spec with rate-limiting headers."

⚖️ Compare

Ask AI to compare approaches within constraints you set. "Compare OT vs CRDT for a doc editor with 10M concurrent documents and offline support — include server coordination requirements."

🔥 Stress-Test

Use AI to attack your own design. "Given the architecture we just designed, enumerate the top 5 bottlenecks and single points of failure, then suggest mitigations for each."

💡
Universal rule: steer before you generate

Before asking AI to generate any artifact, prime it with context: scale numbers, constraints, and the decision you've already made. "Generate the API for my URL shortener" → mediocre output. "I'm using Base62 encoding, Cassandra for the mapping store, Redis with 24h TTL for redirect cache. Generate the REST API spec with idempotency handling and rate-limiting headers" → something worth showing an interviewer.

Chapter 01

The Master Template

An 8-step framework applicable to any system design question — with a clear YOU vs AI split at every stage.

The master template is your repeatable playbook. Every bucket chapter in this guide specializes it for a domain. Internalize the 8 steps and the YOU/AI split — then in the interview, your job is to execute each step while directing AI for the artifact-heavy parts.

The 8 Steps

01

Clarify Scope & NFRs YOU

Before touching any technology, nail the functional and non-functional requirements. Ask: core features (in scope / out of scope), DAU, read/write ratio, latency SLA, consistency requirements, geo-distribution needs, any regulatory constraints. Interviewers reward candidates who don't jump to solutions.

What AI helps with: Once you've stated assumptions, ask AI to "confirm my scope assumptions and flag anything I may have missed for a system of this type."

02

Back-of-Envelope Math YOUAI calculates

You provide: DAU assumption, reads/writes per user per day. AI generates: QPS (average and peak), storage estimates (per record × retention), cache sizing (80/20 rule on hot reads), bandwidth. Then YOU state the design implication — "write-heavy, so optimize write throughput over read latency."

03

Architecture Decision YOU

Monolith vs microservices vs serverless — YOU decide and justify. State your reasoning explicitly: "I'm going microservices because the feed service, user service, and media service have different scaling profiles and independent deployment needs. A monolith would create a deployment bottleneck." Do not ask AI to make this decision — it reveals your judgment (or lack of it).

04

High-Level Component Diagram YOU nameAI generates diagram

YOU name every box: "I need a CDN, an API Gateway, a Feed Service, a User Service, a Timeline Cache (Redis), a Fanout Worker (Kafka consumer), and a Post Store (Cassandra)." Then ask AI to generate the component diagram with data flow arrows, and to describe what protocol connects each box and why.

05

API Design YOU name endpointsAI generates spec

YOU list the endpoint names and the key fields that matter. "I need: POST /posts (create post with media), GET /feed/{userId} (paginated home timeline), POST /follow/{userId} (follow relationship). Cursor-based pagination on the feed." Then ask AI to generate full request/response schemas including auth headers, rate-limit headers, error responses, and idempotency keys for write operations.

06

Data Models YOU define tables & key columnsAI writes DDL

YOU name the entities, their primary keys, partition keys, and any important columns with types. "I need a posts table partitioned by user_id, clustered by created_at DESC, with columns: post_id UUID, user_id UUID, content TEXT, media_urls LIST, like_count COUNTER." Then ask AI to generate the full Cassandra CQL or SQL CREATE statements with indexes, TTLs, and read/write access patterns documented in comments.

07

Sequence Diagram for Critical Flows YOU describe the flowAI generates diagram

Pick the 1–2 hardest flows (e.g., "user creates a post" and "user loads their home timeline"). YOU describe the sequence in plain English — which services are involved, in what order, and where async happens. Ask AI to generate the sequence diagram in Mermaid or text notation, then review it for correctness.

08

Failure Modes & Observability AI enumeratesYOU prioritize

Ask AI: "Enumerate the top failure modes for each tier of the system we designed, in a table: Component | Failure Mode | Detection | Remedy." Then YOU add: which are P0 vs P1, your SLA targets (RPO/RTO), and the key observability metrics (p99 latency, cache hit rate, queue depth, error rate) — because knowing what to monitor signals you've run production systems.

Steering & Correcting AI

🔧 When AI Goes Wrong — What to Say
Wrong DB choice → "That assumes PostgreSQL, but given our write QPS of 50K and need for linear horizontal scale, revise the schema for Cassandra. Use user_id as partition key and created_at as clustering key."
Over-engineered → "That's too complex for our MVP. Assume we start with 1M DAU. Simplify to a single-region deployment and flag which components to add when we reach 100M DAU."
Generic API → "The API spec is too generic. Add: cursor-based pagination using last_post_id, X-RateLimit-Remaining and Retry-After headers, idempotency keys on the POST, and 429/503 error response bodies."
Missing failure modes → "The failure table is missing cache stampede on cold start and Kafka consumer lag. Add both with detection method (Prometheus metric name) and specific remediation steps."
Stress-test prompt → "Now act as a skeptical interviewer. Given the architecture we just designed, what are the top 3 bottlenecks that would appear first under 10× load spike? Propose a mitigation for each."

Bucket 01 — Chapter 02

Social Media & Feed Systems

The fan-out problem, social graphs at scale, and ranking — the hardest read-path problems in distributed systems.

Questions in This Bucket

Design Twitter / X Home Timeline Design Instagram Feed Design Facebook News Feed Design TikTok For You Page Design a Social Graph Service Design YouTube Subscriptions Feed

The core challenge in every feed system is the fan-out problem: when a user posts, how do you efficiently deliver that post to potentially millions of followers? Get this decision right and the rest of the design follows naturally.

What Makes This Bucket Unique

🌟 Celebrity Accounts (Hot Keys)

Accounts with 50M+ followers create extreme write amplification on fan-out-on-write. A single post from @BarackObama would trigger 100M+ cache writes. This breaks naive approaches and forces a hybrid strategy.

🎯 Feed Ranking

Chronological feeds are easy. Relevance-ranked feeds (Instagram, TikTok) require real-time scoring using engagement signals, user affinity, and content preferences — adding an ML inference layer to the hot path.

📹 Media Pipeline

Video posts require async transcoding into multiple resolutions (360p/720p/1080p/4K) before they can be served. The media pipeline is a separate async system with its own failure modes.

📊 Social Graph Storage

Storing who-follows-who at scale: adjacency list in Cassandra (scalable writes) vs graph DB like Neo4j (rich traversal) vs hybrid. The choice affects follow recommendations and feed generation.

What YOU Must Define

✅ You Provide
  • DAU and read/write ratio assumption
  • Fan-out strategy: write / read / hybrid
  • Celebrity threshold (e.g., >1M followers = hybrid)
  • Ranking approach: chronological vs ML-ranked
  • Graph storage choice and justification
  • Media support scope (images / short video / long video)
  • Feed staleness tolerance (real-time vs eventual)
🤖 AI Generates
  • QPS + storage math given your DAU inputs
  • Fan-out comparison table with trade-offs
  • Component diagram with Kafka topology
  • Full API spec (POST /posts, GET /feed, POST /follow)
  • Cassandra DDL for posts + timeline tables
  • Sequence diagram for post creation flow
  • Failure modes table per service tier

Bucket 02 — Chapter 03

Messaging & Real-time Communication

WebSocket management at scale, delivery guarantees for offline users, and group fanout — the hardest real-time problems in distributed systems.

Questions in This Bucket

Design WhatsApp / iMessage Design Slack Design Discord Design Telegram Design a Live Chat (customer support) Design a Notification Service

Messaging systems live or die on delivery guarantees. The hardest problems here are: keeping 100M WebSocket connections alive, delivering messages to offline users with exactly-once semantics, and scaling group fanout when a single channel has 100K+ members.

What Makes This Bucket Unique

🔌 Persistent Connection Management

WebSockets require stateful connections — each server holds a connection pool. This makes horizontal scaling complex (you can't just add servers behind a round-robin LB) and requires session affinity or connection routing.

📬 Offline Delivery

Messages sent while a user is offline must be durably stored and delivered when they reconnect — up to 30 days later. Exactly-once delivery semantics are critical (no duplicate messages on reconnect).

👥 Group Chat at Scale

Group chats with 1000+ members require careful fanout strategy. For large groups, individual message delivery is O(N) — you need pub/sub or topic-based broadcast to handle it efficiently.

🟢 Presence System

Online/away/offline status must propagate to contacts within 2 seconds. This is a high-frequency write problem (heartbeats) that must be batched and aggregated to avoid overloading storage.

What YOU Must Define

✅ You Provide
  • Transport protocol: WebSocket vs long-poll vs SSE
  • Message delivery guarantee: at-most-once, at-least-once, exactly-once
  • Max group size and fanout strategy for large groups
  • Message retention window (e.g., 30 days offline, 1 year history)
  • Message ID strategy (Snowflake ID, UUID, or sequence number)
  • End-to-end encryption scope (yes/no, affects key management)
  • Read receipt semantics (delivered vs read vs seen-by-count)

What Makes This Bucket Unique

🔄 OT vs CRDT

Operational Transformation (OT) requires a central server to order operations. CRDT is peer-to-peer but adds complexity and storage overhead. The choice has profound architectural implications — not just algorithm selection.

⚡ Sub-100ms Collaboration Feel

Users perceive latency above 100ms as "lag" in collaborative tools. This forces local optimistic application of operations before server confirmation — and a rollback mechanism if the server rejects an operation.

🐳 Sandboxed Execution

Online IDEs must execute arbitrary untrusted code in complete isolation. Container escape vulnerabilities, resource exhaustion, and network isolation are not theoretical — they're production incidents.

📝 Version History

Storing every operation for undo/redo and version history at scale (millions of documents) requires careful compaction strategy — you can't store every keystroke forever.

Chapter 06 — Reference

Concepts & Tools

Every technology, algorithm, and pattern referenced in this guide — defined precisely.

Core Algorithms

Fan-out on Write

When a user posts, pre-populate all followers' feed caches immediately. Fast reads, expensive writes. Breaks for celebrity accounts (millions of cache writes per post).

Fan-out on Read

Store post once; compute each user's feed at read time by querying who they follow. Cheap writes, expensive reads. Latency grows with follow count.

Operational Transformation (OT)

Conflict resolution for collaborative editing. Transforms concurrent operations so they produce the same result regardless of arrival order. Requires a central server authority to serialize operations.

CRDT (Conflict-free Replicated Data Type)

Data structures that can be merged from any replica without coordination. Merge is commutative, associative, and idempotent. Enables offline editing and eventual consistency without a central authority.

Token Bucket Algorithm

Rate limiting algorithm. Tokens fill at a fixed rate; each request consumes one token. Allows bursting up to bucket capacity. Implemented atomically in Redis via Lua script.

Consistent Hashing

Distributes data across N nodes such that adding/removing a node only remaps K/N keys (not all keys). Virtual nodes increase even distribution. Used in Redis Cluster, Cassandra, DynamoDB.

Thundering Herd

When a cached item expires, many simultaneous requests all miss cache and hammer the database together. Mitigated by mutex locks, probabilistic early expiration, or request coalescing.

Snowflake ID

Twitter's 64-bit unique ID: 41-bit timestamp + 10-bit machine ID + 12-bit sequence. Time-ordered, generated without coordination. Sortable and compact. Used where UUID is too large or random.

Data Stores

Apache Cassandra

Wide-column NoSQL DB optimized for high write throughput and linear horizontal scale. Data modeled by partition key (determines node) and clustering key (determines sort order within partition). No multi-row transactions.

Redis

In-memory key-value store supporting Strings, Hashes, Lists, Sets, Sorted Sets, Streams. Sub-millisecond latency. Used for caching, session storage, pub/sub, rate limiting (Lua scripts), and leaderboards.

Apache Kafka

Distributed log for high-throughput event streaming. Topics split into partitions for parallelism. Consumer groups allow independent consumption of the same topic. Retention-based (not ack-based) — consumers control their offset.

Elasticsearch

Distributed search engine built on Lucene. Inverted index enables full-text search. Used for search, log analysis, and aggregations. Eventual consistency — not appropriate for transactional data.

Cache Policies

Write-through

Write to cache and DB synchronously. Cache always consistent with DB. Higher write latency. Best for frequently read, frequently updated data (e.g., inventory).

Write-around

Write directly to DB; cache only on first read miss. Cache never stale on writes. Best for write-once, read-rarely data (e.g., product catalog).

Write-back (Write-behind)

Write to cache immediately; async write to DB later. Lowest write latency but risk of data loss on cache failure. Rare in practice for critical data.

LRU Eviction

Least Recently Used — evict the item that was accessed least recently when cache is full. Works well for temporal locality (recently accessed = likely to be accessed again).

Distributed Systems Patterns

Circuit Breaker

Monitors failure rate to a downstream service. After threshold failures, "opens" the circuit — calls fail immediately without attempting the downstream call. Prevents cascading failures. Auto-resets after a probe period.

Idempotency Key

Client-generated unique token attached to a write request. Server uses it to detect and discard duplicate requests. Critical for at-least-once delivery systems where retries are possible.

Dead Letter Queue (DLQ)

A queue that receives messages which failed processing after N retries. Allows manual inspection, debugging, and replay without blocking the main queue.

Cursor-based Pagination

Pagination using a pointer to the last seen item (e.g., last_id, last_score+last_id). Stable as items are inserted/deleted. O(1) vs offset pagination's O(N) scan. Required for real-time feeds.

API & Protocol

WebSocket

Full-duplex TCP connection after HTTP upgrade handshake. Persistent — both sides can send at any time. Used for real-time features (chat, collaborative editing, live feed updates).

Server-Sent Events (SSE)

One-way HTTP streaming from server to client. Simpler than WebSocket (no upgrade). Good for live updates where client doesn't need to send data (dashboards, notifications). Not for bidirectional communication.

Base62 Encoding

Encodes numbers using 62 characters [0-9A-Za-z]. Compact URL-safe representation. 7 Base62 characters represent 62^7 = 3.5 trillion unique values. Used in URL shorteners.

cgroups v2

Linux kernel feature for resource isolation: limit CPU, memory, PIDs, I/O per process group. Foundation of Docker container resource enforcement. pids.max prevents fork bombs.

NFRs & SLA Terminology

RPO (Recovery Point Objective)

Maximum acceptable data loss measured in time. RPO=1hr means you can lose up to 1 hour of data. Drives backup frequency and replication lag tolerance.

RTO (Recovery Time Objective)

Maximum acceptable downtime after a failure. RTO=30s means the system must be back online within 30 seconds. Drives auto-failover speed and replica promotion time.

QPS (Queries Per Second)

Request throughput metric. Average QPS = DAU × requests_per_user / 86,400. Peak QPS = 2–3× average. Always design for peak, not average.

DAU (Daily Active Users)

The primary scale anchor. All back-of-envelope estimates derive from DAU. State your DAU assumption explicitly before any math — it sets the scale of every subsequent number.

Meta Questions About the AI-Assisted Format

Why are you asking AI to generate the API spec instead of writing it yourself?
Model Answer: "I'm using AI to accelerate the artifact generation, not to make design decisions. I've already decided the endpoint names, the authentication mechanism, cursor-based pagination, and idempotency handling — those are engineering judgment calls that require domain knowledge. Generating the JSON schema and HTTP headers from those specifications is a mechanical task where AI is faster and more consistent than manual writing. It frees my time to focus on the architectural trade-offs, which is where the actual engineering thinking lives."
The AI suggested a different database than what you chose. Why did you override it?
Model Answer: "AI tools tend to default to SQL databases because they appear in more training examples. But for this access pattern — 50K writes/second, partitioned by user, with no multi-row transaction requirements — Cassandra is the right choice. It gives us linear horizontal write scalability, which PostgreSQL can't match without significant sharding complexity. I steered the AI back to Cassandra and asked it to revise the schema to use user_id as the partition key with created_at as the clustering key, which gives O(1) writes and O(1) reads for the latest feed."
How do you know if the output the AI generated is correct?
Model Answer: "Two ways: first, I review every artifact against my mental model — I know what a correct Cassandra schema looks like for this access pattern, so I can spot when partition keys or clustering columns are wrong. Second, I use AI to stress-test its own output: I ask 'what are the top 3 issues with this design?' and review those critiques. If AI can't find issues I know exist, that tells me something about the quality of the output. The reviewer mindset is the same one I'd apply to a code review from a junior engineer."

System Design Follow-up Questions

Your feed system uses hybrid fan-out. What happens when a non-celebrity user suddenly goes viral?
Model Answer: "A user in the write-fanout tier gaining 5M followers overnight creates a window where they're still treated as a regular user but generating celebrity-scale writes. The mitigation is a background job that monitors follower growth rate — if a user gains 100K followers in 24 hours, they're promoted to the celebrity tier immediately. The Fanout Worker checks the celebrity-accounts Redis SET on every post; once promoted, new posts use read-fanout. Existing pre-computed timelines remain valid until they expire (TTL-based). This is a real incident type — Twitter had exactly this problem with breaking news accounts."
Your WebSocket design has Chat Servers with stateful connections. How do you deploy a new version without dropping connections?
Model Answer: "Zero-downtime WebSocket deploys require a drain strategy. The deployment process: (1) Mark the old pod as draining — stop accepting new connections via health check endpoint returning 503; (2) New pods come up, accepting new connections; (3) Old pod broadcasts a reconnect signal to its connected clients — clients receive it and reconnect to the load balancer, which routes them to a new pod; (4) After a drain window (say 60 seconds), the old pod forcibly closes remaining connections and terminates. Client-side reconnect logic is critical — clients must handle disconnection gracefully, buffer pending messages, and reconnect with their last_received_message_id for seamless replay."
Your rate limiter uses Redis. At 5M requests/second, is Redis the bottleneck?
Model Answer: "Yes, centralized Redis starts becoming a bottleneck around 1-2M operations/second per node. At 5M req/sec, we need either a Redis Cluster (shard rate limit keys by user_id hash) or a two-tier approach: L1 local token bucket on each app server (enforces ~2× the limit, loose but fast at ~nanosecond latency) + L2 Redis for global accuracy. The L1 tier handles the vast majority of requests locally and only syncs with Redis periodically (every 100ms) to compensate for overages. This is how Stripe and Cloudflare implement their rate limiters at scale — the local tier absorbs >99% of checks."
How would you add end-to-end encryption to your messaging system?
Model Answer: "E2E encryption fundamentally changes the architecture. With E2E encryption, the server never sees plaintext — messages are encrypted client-side with the recipient's public key and decrypted only on the recipient's device. This means: (1) The server can't read or search message content; (2) Message notifications need to include only metadata (sender, timestamp) not content; (3) Key management becomes critical — each user has a key pair, public keys are stored server-side, private keys never leave the device. The tricky part is multi-device support: a user's second device needs to decrypt historical messages. WhatsApp's Signal Protocol uses a key ratchet mechanism — each message uses a new encryption key derived from the previous one, so compromise of one key doesn't expose past messages."
Walk me through how you'd monitor this system in production. What are your first 5 alerts?
Model Answer: "The five alerts I'd set up first, in priority order: (1) API p99 latency > 500ms for 2 consecutive minutes — catches the broadest class of user-facing degradation; (2) Error rate > 1% on any endpoint — catches failures that might look like success from infrastructure metrics; (3) Kafka consumer lag > 100K messages on the fanout topic — catches processing backlog before it becomes user-visible feed delay; (4) Redis cache hit rate < 85% for 5 minutes — usually signals a cold-start event or thundering herd; (5) Database replication lag > 500ms — means read replicas are serving stale data. I'd set these up before we go live — not in response to incidents. The goal is to know about degradation before users do."
