Spring Boot

Autoconfiguration, dependency injection, production-readiness, and scaling patterns for backend services.

AutoconfigurationDI / IoCActuator Spring SecuritySpring DataTransactions Async / ReactiveProfiles

Spring Boot is not just a framework — it's an opinionated production platform. A staff engineer must understand the full lifecycle: how autoconfiguration works, how the IoC container manages beans, how transactions behave, and where the sharp edges are at scale.

Autoconfiguration & the IoC Container

Spring Boot's autoconfiguration works via @EnableAutoConfiguration, which reads META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (Boot 3.x). Each autoconfiguration class is conditional — it only fires when specific classes are on the classpath (@ConditionalOnClass) or beans are absent (@ConditionalOnMissingBean).

Mental Model

Autoconfiguration is like a smart factory that wires up machines based on what raw materials it finds. If it sees a JDBC driver on the classpath, it auto-creates a DataSource. If you supply your own, it steps aside. You stay in control by overriding; the factory handles defaults.

Bean Scopes — Production Implications

Singleton (Default)

One instance per Spring context. Shared across all threads. Fastest but requires thread-safe state. Injecting prototype beans into singletons requires ObjectProvider or method injection to avoid stale instances.

Prototype

New instance per injection point. Spring does not manage the full lifecycle (destroy never called). Use for stateful objects. Common mistake: injecting into singleton causes only one prototype to ever be created.

Request / Session

Web-only scopes. @RequestScope bean tied to HTTP request lifecycle. Useful for per-request context (e.g., user principal caching). Requires ScopedProxyMode when injecting into longer-lived beans.

Custom Scopes

Implement org.springframework.beans.factory.config.Scope. Used for tenant-scoped or conversation-scoped beans in multi-tenant SaaS applications. Rarely needed but shows depth.

Transaction Management

Spring's declarative transactions via @Transactional proxy calls through AOP. This means self-invocation breaks transactionality — calling a @Transactional method from within the same class bypasses the proxy.

Spring Security Architecture

The security filter chain intercepts every request. Key filters in order: SecurityContextPersistenceFilter → UsernamePasswordAuthenticationFilter → ExceptionTranslationFilter → FilterSecurityInterceptor.

JWT with Spring Security: Replace form-based authentication with a custom OncePerRequestFilter that validates JWTs and populates the SecurityContext. Use method security (@PreAuthorize, @PostAuthorize) for fine-grained access control.

Spring Boot Actuator for Production

Actuator endpoints are the primary observability surface for Spring Boot apps in production. Key endpoints: /actuator/health, /actuator/metrics, /actuator/env, /actuator/threaddump, /actuator/heapdump.

Always secure actuator endpoints — expose only /health and /info publicly, others behind an internal port or auth. In GCP, expose metrics via Micrometer → Cloud Monitoring integration.

Async & Reactive Patterns

@Async runs tasks on a TaskExecutor thread pool. Always configure a named executor — default is SimpleAsyncTaskExecutor which creates a new thread per task (catastrophic at scale). Reactive via Spring WebFlux (Project Reactor) provides non-blocking I/O on Netty — ideal for high-concurrency I/O-bound workloads. Not a silver bullet for CPU-bound work.

JWT-based stateless authentication setup

@Component
public class JwtAuthFilter extends OncePerRequestFilter {
  private final JwtService jwtService;
  private final UserDetailsService userDetailsService;

  @Override
  protected void doFilterInternal(HttpServletRequest req,
      HttpServletResponse res, FilterChain chain)
      throws ServletException, IOException {

    String header = req.getHeader("Authorization");
    if (header == null || !header.startsWith("Bearer ")) {
      chain.doFilter(req, res); return;
    }
    String token = header.substring(7);
    String username = jwtService.extractUsername(token);

    if (username != null && SecurityContextHolder.getContext()
        .getAuthentication() == null) {
      UserDetails user = userDetailsService.loadUserByUsername(username);
      if (jwtService.isTokenValid(token, user)) {
        var authToken = new UsernamePasswordAuthenticationToken(
            user, null, user.getAuthorities());
        authToken.setDetails(new WebAuthenticationDetailsSource()
            .buildDetails(req));
        SecurityContextHolder.getContext().setAuthentication(authToken);
      }
    }
    chain.doFilter(req, res);
  }
}

The N+1 Problem & DataLoader

The N+1 problem occurs when resolving a list of N items each triggers an individual database query for a child field — resulting in 1 + N queries instead of 2.

Mental Model

Imagine fetching a list of 100 orders and then separately querying for each order's customer. Without batching, that's 100 individual DB calls. DataLoader is the solution: it collects all 100 customer IDs across a request tick, fires one batched query, then distributes the results back to each resolver.

DataLoader Pattern: Spring for GraphQL integrates with dataloader. You register a BatchLoaderRegistry per entity type. Requests within the same "tick" are automatically batched.

Schema Design Best Practices

  • Design for the client, not the database. GraphQL types should reflect what clients need, not mirror table schemas.
  • Use Connections for pagination (Relay spec): UserConnection { edges { node cursor } pageInfo }
  • Prefer nullable fields conservatively. Non-null (!) types are a contract — if they ever return null, the entire parent field nulls out, which can cascade up the response tree.
  • Union vs Interface: Use interfaces for shared fields; unions for heterogeneous types with no common fields.
  • Mutations should be fat — accept a full input type, return the modified object. Avoid fine-grained mutations per field.

Security Concerns

Disable introspection in production — the schema is a roadmap for attackers. Use persisted queries (only allow pre-registered query hashes) for public-facing APIs. Apply query depth limits and query complexity analysis to prevent DoS via deeply nested queries. Rate-limit by token/IP at the gateway level.

Federation & Micro-Frontends

GraphQL Federation (Apollo Federation or Spring's equivalent) allows splitting a schema across services. Each subgraph owns a slice; the gateway stitches them at query time. Key concepts: @key directive marks entity identifiers, @extends allows subgraphs to contribute fields to another service's type.

// Register a batch loader for User entities
@Configuration
public class DataLoaderConfig {
  @Bean
  public RuntimeWiringConfigurer batchLoaderWiring(UserRepository repo) {
    return builder -> builder.dataFetcher("Order", "customer",
      env -> {
        DataLoader<String, User> loader =
          env.getDataLoader("userLoader");
        String userId = env.getSource().getUserId();
        return loader.load(userId); // batched across the request tick
      });
  }

  @Bean
  public BatchLoaderRegistry batchLoaderRegistry(UserRepository repo) {
    return registrar -> registrar
      .forTypePair(String.class, User.class)
      .withName("userLoader")
      .registerBatchLoader((ids, env) ->
        Flux.fromIterable(repo.findAllById(ids))); // ONE query for all IDs
  }
}

Cloud Spanner & GCP

Globally distributed ACID transactions, schema design, hotspot avoidance, and GCP observability at scale.

TrueTimeInterleavingHotspot Avoidance Read StalenessCloud ArmorPub/Sub Cloud RunGKESecret Manager

Cloud Spanner is not just a scalable relational database — it's a horizontally scalable, externally consistent database that redefines what's possible with distributed ACID transactions. Understanding its internals is what separates a developer who uses it from an architect who designs with it.

Why Spanner Is Different

Traditional distributed DBs sacrifice either consistency (eventual consistency) or availability (CAP theorem). Spanner achieves external consistency (stronger than serializability) by using TrueTime — atomic clocks and GPS in every datacenter — to assign globally ordered commit timestamps. This means transactions across regions are linearizable.

Mental Model

Imagine a bank with branches in New York, London, and Tokyo. Traditional distributed DBs require you to accept that a transfer seen in NY might not be visible in Tokyo immediately. Spanner's TrueTime is like having a single global atomic clock so that every branch sees exactly the same transaction order — worldwide, always.

Schema Design — Hotspot Avoidance

Spanner splits data into splits (like shards) based on key range. Sequential primary keys (e.g., AUTO_INCREMENT integers) create a "hotspot" — all inserts land on the same split. This is the most common Spanner performance mistake.

Solutions:

  • UUID v4 as primary key — random distribution across splits
  • Bit reversal of sequential integers
  • Hash prefix prepended to sequential ID
  • Timestamp-based sharding: prefix with FARM_FINGERPRINT(GENERATE_UUID())

Interleaved Tables

Interleaving co-locates child rows with parent rows on the same split. Reading a parent with all its children becomes a local operation with no cross-split joins — dramatically faster for hierarchical data. Parent key must be the prefix of child's primary key.

Read Staleness

Spanner offers two read modes:

  • Strong reads — always read the latest committed data. Higher latency (must wait for quorum).
  • Stale reads — read data as of a timestamp in the past (e.g., 15 seconds ago). Much lower latency, no quorum needed. Safe for analytics, dashboards, read-heavy GET endpoints where slight staleness is acceptable.

Staff insight: use stale reads for 80% of your API reads. This dramatically reduces read latency and Spanner compute usage with no perceptible user impact.

GCP Architecture Patterns

Cloud Run vs GKE

Cloud Run: fully managed, scales to zero, per-request billing. Ideal for bursty or stateless APIs. GKE: full Kubernetes, needed for stateful workloads, complex networking (service mesh), or when you need DaemonSets/sidecar patterns. Spring Boot works well on both.

Secret Manager

Store DB credentials, API keys, and JWT secrets in Secret Manager. Inject into Spring Boot via spring-cloud-gcp-starter-secretmanager. Avoid environment variables for secrets — they appear in process lists and K8s pod specs in plaintext. Use Workload Identity to bind service accounts to pods.

Pub/Sub for Async

Use Pub/Sub for event-driven decoupling. Spring Cloud GCP provides PubSubTemplate and @PubSubListener. Key considerations: exactly-once delivery requires idempotent consumers. Use message attributes for filtering; don't encode routing in the topic name.

Cloud Armor

WAF + DDoS protection at the load balancer level. Configure security policies for SQL injection, XSS detection, rate limiting per IP. For GraphQL, add a custom rule to reject requests with excessively large bodies (query bombs).

        .ofExactStaleness(15, TimeUnit.SECONDS);

      .executeQuery(Statement.of("SELECT * FROM products"))
      .stream().map(this::toProduct).toList();

Questions You May Be Asked

How does Cloud Spanner achieve external consistency across regions?
Expected: TrueTime API — atomic clocks + GPS gives bounded clock uncertainty. Commits wait for the uncertainty interval to pass (commit-wait) so that the assigned timestamp is guaranteed to be in the past globally. This enables serializability across geographies without the usual distributed coordination penalty.
You're designing a schema for a high-write event stream in Spanner. What are your considerations?
Expected: Hotspot avoidance (no sequential PKs), shard_id prefix for range queries, allow_commit_timestamp for ordering, partitioned DML for bulk deletes, TTL policies. Discuss split boundaries and how Spanner automatically splits hot ranges.
When would you use Pub/Sub vs direct Spanner writes for an event-sourcing pattern?
Expected: Pub/Sub for decoupling producers from consumers, fan-out to multiple downstream systems, buffering bursts. Direct Spanner write for transactional outbox patterns (event + state change in same transaction). Outbox pattern avoids dual-write inconsistency.
⚡

Apollo Client & GraphQL Integration

Apollo Client's normalized cache stores every object by __typename:id. This means a mutation that returns an updated object automatically updates every query that contains that object — no manual cache invalidation needed for simple cases.

Common cache patterns:

  • Optimistic updates: Update the cache before the mutation completes for instant UI feedback. Roll back on error.
  • Cache eviction: After operations that affect list ordering (e.g., sort by date after insert), use cache.evict + cache.gc to invalidate stale list queries.
  • Field policies: Custom merge functions for paginated fields using Relay-style connections. Essential for infinite scroll.

Render Performance

  • useMemo / useCallback: Only wrap computations/functions that are genuinely expensive or passed to memoized children. Overuse creates cognitive overhead with no perf benefit.
  • React.memo: Prevents re-renders when parent re-renders but props haven't changed. Only valuable when combined with stable references (useCallback).
  • Virtualization: For lists of 100+ items, use react-window or react-virtual to render only visible items.
  • Code splitting: React.lazy + Suspense for route-level splitting. Every route should be its own bundle chunk.

React Server Components (RSC) — Staff-Level Understanding

RSC runs on the server and sends HTML (not JS) to the client — zero bundle size for the component. Ideal for data-fetching components that don't need interactivity. Cannot use hooks or browser APIs. In a Next.js + Spring Boot + GraphQL architecture, RSC can query your GraphQL endpoint directly at the server, sending ready HTML to the client with no client-side round-trip needed.

Unit Testing & Aggregation vs Inheritance

Unit testing and OOP design were asked in your last round — which means the panel values engineering foundations. Your answers should go beyond definitions: show the reasoning, the failure modes, and how these principles affect real architecture decisions.

Unit Testing — The Right Mental Model

A unit test verifies a unit of behaviour, not a unit of code. One test class per method is not the goal — testing all meaningful observable behaviours of a component in isolation is. Good unit tests follow FIRST:

F — Fast

Tests run in milliseconds. If a test needs a real DB or HTTP call, it's an integration test. Use mocks/stubs for I/O. Slow tests get skipped under deadline pressure — the worst outcome.

I — Isolated

Each test sets up its own state. No shared mutable state between tests. Order-independent. @BeforeEach should reset all mocks and re-create the SUT. Tests that fail based on execution order are time bombs.

R — Repeatable

Same result on every machine, every run. No dependency on system clock (use Clock injection), no random data without seeding, no external service calls without mocking.

S — Self-validating

Binary pass/fail. No manual inspection of output. Assert all meaningful state changes and outputs. Use assertj for fluent, readable assertions — not just assertTrue(result != null).

Aggregation vs Inheritance

Inheritance (extends) creates an "is-a" relationship. The subclass gets the parent's interface and implementation. Aggregation (composition, "has-a") creates an object that uses another object through a reference.

⚠️
The Fragile Base Class Problem When you inherit from a class, you couple tightly to its implementation — not just its interface. A change to the base class can silently break subclasses. This is the strongest argument for preferring composition: the composed dependency is a black box; only its interface contract matters.

When to use inheritance: True "is-a" relationships where the subtype must pass the Liskov Substitution Principle (LSP) — every subclass must be substitutable for its parent without breaking callers. Classic example: Exception hierarchy, InputStream hierarchy.

When to use composition (default choice): Reusing behaviour, varying behaviour at runtime (strategy pattern), avoiding deep hierarchies, following the Open/Closed Principle without modifying base classes.

Scalability & DB Constraints

Horizontal scaling principles, how they interact with database-layer constraints, and the patterns that bridge stateless services with stateful persistence.

Horizontal ScalingCAP TheoremIdempotency Distributed LocksCQRSSharding Connection PoolingOutbox Pattern

Horizontal scaling of stateless services is easy — add instances behind a load balancer. The hard part is what happens to your database, your sessions, your distributed state, and your transactional guarantees when you do. This is where staff engineers think differently.

The Core Tension

Adding more application instances creates stateless compute scale. But the database is stateful — it becomes the bottleneck. Every scaling decision at the application tier must be analyzed for its database impact.

Problem: Distributed Session State

If each app instance stores session in memory, a user's request may hit a different instance and lose their session. Solutions: Sticky sessions (routing, but single point of failure), or externalize session to Redis/Spanner (stateless instances, any can handle any request).

Problem: Race Conditions Under Scale

With N instances, concurrent requests can cause inventory oversell, double-charge, or duplicate inserts that weren't possible with a single instance. Solutions: optimistic locking (version column), database-level unique constraints, idempotency keys.

Problem: Connection Pool Exhaustion

With 20 instances × 20 pool connections = 400 DB connections. Spanner handles this well (sessions are multiplexed), but traditional DBs can choke. Solutions: connection pooler (PgBouncer), async/reactive stack to reduce threads-per-connection ratio.

Problem: Unique Constraint Violations

Two instances concurrently try to INSERT the same record (e.g., creating user with same email). DB UNIQUE constraint is the final guardrail — catch DataIntegrityViolationException and handle gracefully. Application-level checks are not reliable without locks.

Idempotency — The Key Pattern for Scale

An idempotent operation produces the same result no matter how many times it's called. Critical for horizontal scaling because retries are inevitable (network failures, load balancer timeouts). Design every mutating API to accept an idempotency key:

  1. Client generates a UUID and attaches it as a header (Idempotency-Key: uuid)
  2. Server stores the key + response in Spanner with a TTL
  3. On duplicate request: return cached response immediately, no re-execution

The Outbox Pattern

A critical pattern for publishing events to Pub/Sub reliably from a horizontally scaled service: write to a Spanner outbox table in the same transaction as your business data. A separate poller reads unprocessed outbox rows and publishes to Pub/Sub, then marks them as published. This avoids dual-write inconsistency (where the DB write succeeds but the Pub/Sub publish fails).

Idempotency key enforcement in Spring

@Component
public class IdempotencyFilter extends OncePerRequestFilter {
  private final IdempotencyKeyRepository repo;

  @Override
  protected void doFilterInternal(HttpServletRequest req,
      HttpServletResponse res, FilterChain chain)
      throws ServletException, IOException {

    String key = req.getHeader("Idempotency-Key");
    if (key == null) { chain.doFilter(req, res); return; }

    Optional<CachedResponse> cached = repo.findByKey(key);
    if (cached.isPresent()) {
      // Return stored response without re-executing
      res.setStatus(cached.get().getStatus());
      res.getWriter().write(cached.get().getBody());
      return;
    }

    // Wrap response to capture it for caching
    CachingResponseWrapper wrapper = new CachingResponseWrapper(res);
    chain.doFilter(req, wrapper);

    // Store result — expires in 24h
    repo.save(new CachedResponse(key, wrapper.getStatus(),
      wrapper.getCapturedBody(), Instant.now().plus(24, HOURS)));
  }
}

Security: PII Handling & Password Hashing

How to protect sensitive user data at rest and in transit, the right way to hash passwords, and PII governance at scale.

bcrypt / Argon2SaltingData Classification Encryption at RestField-Level Encryption GDPR / CCPAData MaskingAudit Logging

Security questions are often asked to gauge judgment, not just knowledge. The key is understanding why certain approaches are correct — what attacks they prevent — and where the residual risk lies even after mitigation.

Password Hashing — The Full Answer

Never store plaintext passwords. Never use MD5 or SHA-256 alone for passwords. MD5/SHA-256 are fast hash functions designed for checksums/data integrity — their speed is a security liability for passwords because it enables fast brute-force attacks.

Password hashing requires slow, memory-hard algorithms designed specifically to resist GPU-accelerated brute force:

bcrypt

Industry standard since 1999. Work factor (cost parameter) is adjustable — increase it as hardware speeds up. Built-in salt. Spring Security's BCryptPasswordEncoder handles everything. Default cost factor 10 ≈ 100ms per hash.

Argon2id

Winner of Password Hashing Competition (2015). Best choice for new systems. Memory-hard (resist GPU/FPGA attacks). Three parameters: time cost, memory cost, parallelism. argon2-jvm library for Java. OWASP recommendation as of 2024.

scrypt

Memory-hard predecessor to Argon2. Still acceptable. More tunable than bcrypt. Less widely supported in language ecosystems than bcrypt/Argon2id. Use Argon2id for new systems.

Why Salting Matters

A salt is a random value prepended to the password before hashing. Purpose: prevents rainbow table attacks and ensures that two users with the same password produce different hash values. bcrypt and Argon2 generate and embed the salt automatically — you don't manage salts manually.

PII Handling — The Governance Framework

Data Classification first: Not all PII is equal. Define tiers:

  • Tier 1 — Highly Sensitive: SSN, financial account numbers, health records, passwords. Always encrypted at field level, access requires explicit justification.
  • Tier 2 — Sensitive: Email, phone number, home address, date of birth. Encrypted at rest, access logged.
  • Tier 3 — Internal: User ID, order history, behavioral data. Access controlled but not field-level encrypted.

Technical Controls

Encryption at Rest

GCP encrypts Spanner data at rest by default (Google-managed keys). For additional control: CMEK (Customer-Managed Encryption Keys) via Cloud KMS. For field-level encryption (FLE), encrypt before writing to Spanner using AES-256-GCM with KMS-managed keys.

Encryption in Transit

TLS 1.2+ mandatory for all service-to-service calls. In GCP: service mesh (Istio on GKE) provides mTLS automatically. For Cloud Run: Google manages TLS. Spanner Java client uses TLS by default.

Data Masking in Logs

Never log raw PII — phone numbers, emails, SSNs. Use log masking filters. In Spring Boot: customize Jackson serializer to mask fields annotated with @Sensitive. Structured logging (JSON) with field-level masking is easier to audit than free-text logs.

Right to Erasure (GDPR)

Design for data deletion: pseudonymization (replace real values with UUID, store mapping in a crypto-erase table). Delete the mapping key → all references become meaningless. More practical than finding and deleting all copies of PII scattered across tables and event streams.

Audit Logging for PII Access

Every access to Tier 1/2 PII must be logged: who accessed it, when, from what service/endpoint, for what purpose. Use GCP Cloud Audit Logs (Data Access audit logs) at the Spanner level, and application-level audit events published to Pub/Sub → BigQuery for analysis.

  public String encrypt(String plaintext) throws Exception {

  }

Monitoring is not just dashboards — it's your feedback loop for production systems. At staff level, the question is never "are you monitoring?" but "are you monitoring the right things, acting on the right signals, and using observability to drive engineering decisions?"

The Three Pillars of Observability

📊 Metrics

Aggregated numeric measurements over time. Low storage cost. Great for alerting thresholds and dashboards. Micrometer → Cloud Monitoring. KEY: track business metrics alongside technical metrics (orders/min, not just HTTP req/s).

📋 Logs

Detailed event records. High storage cost. Essential for debugging individual failures. Structured JSON logging (Logback + GCP logging) enables querying. Never log PII. Correlate with trace IDs for request-level debugging.

🔍 Traces

Distributed request traces across services. Spring Cloud Sleuth (Micrometer Tracing in Boot 3) adds trace/span IDs. Cloud Trace visualises end-to-end latency. Identifies which service or Spanner query is causing p99 latency spikes.

SLI / SLO / SLA — The Framework

ConceptDefinitionExample
SLI (Indicator)The metric being measuredPercentage of HTTP requests returning 2xx in ≤ 200ms
SLO (Objective)Target value for the SLI — internal commitment99.9% of requests succeed in ≤ 200ms, measured over 30 days
SLA (Agreement)External contract with consequences for breach99.5% uptime guaranteed; credits issued if violated
Error BudgetAllowed failures before SLO breach (100% - SLO)0.1% = 43.8 min/month of allowed downtime

Error Budget Decision Framework: If error budget is healthy → ship features aggressively. If error budget is burning → freeze releases, prioritize reliability work. This removes blame from reliability conversations and makes it data-driven.

The Four Golden Signals (Google SRE)

  • Latency: Time to serve a request. Track p50, p95, p99 — averages hide tail latency issues.
  • Traffic: Demand on the system. Requests/sec, queries/sec, messages/sec.
  • Errors: Rate of failed requests. Distinguish 5xx (server errors) from 4xx (client errors).
  • Saturation: How full the system is. CPU %, memory %, thread pool queue depth, DB connection pool utilization.

Alerting Philosophy

Alert on symptoms, not causes. Alert on high error rate (what users experience), not CPU spike (internal state that may or may not matter). Avoid alert fatigue — every alert must be actionable. If an alert fires and the on-call engineer has no clear action to take, remove or downgrade the alert.

Alert levels: Page (PagerDuty, immediate action) → Ticket (actionable next business day) → Dashboard-only (informational, monitor trends).

A test suite is a production system. It has maintenance cost, performance characteristics, false positive rates, and architectural debt. Staff engineers think about the ROI of tests, not just coverage numbers.

The Testing Pyramid — and Why It's a Guideline, Not a Law

The classic pyramid: many unit tests at the base, fewer integration tests in the middle, few E2E tests at the top. The reasoning: unit tests are fast, cheap, and isolated; E2E tests are slow, expensive, and brittle. But the right distribution depends on your system:

  • Unit tests (70%): Business logic, domain rules, edge cases, error handling. No I/O. Fast.
  • Integration tests (20%): Controller + service + repository layer. Spanner via Testcontainers emulator, or Spring Data JPA with H2. Slower but tests real wiring.
  • E2E / Contract tests (10%): Critical user journeys (checkout, auth), API contract validation with Pact. Don't test every flow E2E.

Test Coverage — The Right Philosophy

Line coverage is a vanity metric. Branch coverage is more meaningful. Mutation testing (Pitest) is the most rigorous: it mutates your code (e.g., flips a condition) and checks if any test fails. If no test catches the mutation — your tests are not actually verifying that logic.

Coverage targets: Don't mandate 80% line coverage. Instead mandate: all business rules have tests, all exception paths have tests, all public API contracts have tests. Let coverage reports guide where tests are missing — not as a compliance checkbox.

Contract Testing with Pact

In a microservices architecture with GraphQL, contract tests verify that the provider (Spring Boot API) still satisfies the consumer's (React app) expectations after code changes. Pact generates a contract file from consumer tests and verifies it against the provider — catching breaking schema changes before deployment.

Java & Spring Boot Best Practices

Modern Java idioms, Spring Boot production patterns, and the opinionated choices that signal staff-level engineering judgment.

RecordsSealed ClassesOptional StreamsVirtual ThreadsLayered Architecture Exception HandlingConfiguration Properties

Best practices are not rules to recite — they're trade-offs to justify. When asked about Java or Spring Boot practices, always explain the why, not just the what. That's what separates staff-level answers from senior-level answers.

Modern Java (17–21) Idioms

Records (Java 16+)

Immutable data carriers. Auto-generates constructor, getters, equals, hashCode, toString. Use for DTOs, value objects, response/request types. Cannot extend another class (but can implement interfaces). Compact canonical constructor for validation.

Sealed Classes (Java 17+)

Restrict which classes can extend a class. Powerful with pattern matching switch. Use for domain result types: sealed interface Result permits Success, Failure. Enables exhaustive switch — compiler errors if a new subtype is added and switch isn't updated.

Virtual Threads (Java 21)

Lightweight threads managed by JVM. Millions can exist simultaneously. Makes blocking I/O (Spanner calls, HTTP calls) viable at high throughput without reactive programming. Spring Boot 3.2+ supports virtual threads with one config line: spring.threads.virtual.enabled=true.

Pattern Matching

instanceof with binding: if (obj instanceof String s) { ... } — no cast needed. Switch pattern matching (Java 21): exhaustive, type-safe switch over sealed hierarchies. Eliminates large if-instanceof chains.

Optional — Correct Usage

Use Optional as a return type for methods that may return no value. Never use it for: fields, constructor parameters, collections (empty collection > Optional<List>), method parameters (use overloads instead). Never call .get() without .isPresent() — use orElseThrow() or map() instead.

Exception Handling Strategy

Follow a clear exception hierarchy. Checked exceptions should only be used when the caller has a realistic recovery path (rare). Use unchecked exceptions (RuntimeException) for programming errors and domain violations. Have a centralized @ControllerAdvice that maps domain exceptions to HTTP responses — don't scatter error-to-HTTP mapping across controllers.

Configuration Best Practices

Avoid @Value on individual fields for complex configs — use @ConfigurationProperties with a bound POJO. This enables: validation (@Validated), autocomplete in IDEs, documentation, and refactoring safety. Group config by concern — never have 50 individual @Value annotations in a service.

Layered Architecture

Strict layering with clear dependency direction: Controller → Service → Repository. Never call a Repository directly from a Controller. Never have business logic in a Controller. Services should return domain objects, not JPA entities (use mappers). This ensures: testability, separation of HTTP concerns from business logic, database abstraction.

Centralized exception handling with @ControllerAdvice

@ControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

  @ExceptionHandler(NotFoundException.class)
  public ResponseEntity<ErrorResponse> handleNotFound(
      NotFoundException ex, HttpServletRequest req) {
    return ResponseEntity.status(404)
      .body(new ErrorResponse("NOT_FOUND", ex.getMessage(),
        req.getRequestURI(), Instant.now()));
  }

  @ExceptionHandler(PaymentDeclinedException.class)
  public ResponseEntity<ErrorResponse> handlePaymentDeclined(
      PaymentDeclinedException ex) {
    log.warn("Payment declined for user {}: {}", ex.getUserId(), ex.getReason());
    return ResponseEntity.status(422)
      .body(new ErrorResponse("PAYMENT_DECLINED", ex.getReason(), null, Instant.now()));
  }

  @ExceptionHandler(MethodArgumentNotValidException.class)
  public ResponseEntity<ValidationErrorResponse> handleValidation(
      MethodArgumentNotValidException ex) {
    var errors = ex.getBindingResult().getFieldErrors().stream()
      .map(fe -> new FieldError(fe.getField(), fe.getDefaultMessage()))
      .toList();
    return ResponseEntity.badRequest()
      .body(new ValidationErrorResponse("VALIDATION_FAILED", errors));
  }

  @ExceptionHandler(Exception.class)
  public ResponseEntity<ErrorResponse> handleUnexpected(Exception ex) {
    log.error("Unhandled exception", ex);  // never expose stack trace to client
    return ResponseEntity.internalServerError()
      .body(new ErrorResponse("INTERNAL_ERROR",
        "An unexpected error occurred", null, Instant.now()));
  }
}

Concepts & Tools Reference

Quick-reference definitions for every tool, pattern, and concept mentioned in this guide.

A glossary to confirm vocabulary and review quick definitions before the interview. Concepts are listed alphabetically by category.

Spring Ecosystem

AutoconfigurationSpring Boot mechanism that automatically creates beans based on classpath contents and existing bean definitions. Controlled via @ConditionalOn* annotations. Readable from imports file at META-INF/spring.
@TransactionalAOP-based proxy that wraps method execution in a database transaction. Self-invocation bypasses the proxy. Rollback defaults to unchecked exceptions only.
@ConfigurationPropertiesBinds a prefix of properties file entries to a typed POJO. Supports validation. Preferred over @Value for complex configuration.
ActuatorSpring Boot module exposing operational endpoints: health, metrics, env, threaddump. Integrates with Micrometer for metric export.
MicrometerVendor-neutral metrics facade for JVM applications. Backends: Cloud Monitoring (Stackdriver), Prometheus, Datadog. Analogous to SLF4J for metrics.
Spring Security Filter ChainOrdered chain of servlet filters that handle authentication and authorization. Custom filters added via SecurityFilterChain @Bean configuration.
Virtual Threads (Java 21)Lightweight JVM-managed threads that yield on blocking I/O. Enables high-concurrency blocking code without reactive programming overhead. Enabled in Spring Boot 3.2+ with a single config property.

GraphQL

DataLoaderBatching and caching utility that collects individual load requests within a single request tick and fires one batched query. Solves the N+1 problem in GraphQL resolvers.
N+1 ProblemPerformance antipattern where fetching N parent items triggers N additional queries for child items. Result: 1+N DB queries instead of 2.
FederationArchitecture pattern where multiple GraphQL services each own a subgraph. A gateway stitches them into a unified schema. Apollo Federation is the most common implementation.
Persisted QueriesTechnique where client pre-registers queries by hash with the server. Only registered query hashes are accepted in production — prevents arbitrary query execution and introspection-based attacks.
Query ComplexityNumeric cost assigned to a query based on its structure and field costs. Queries exceeding a threshold are rejected before execution. Prevents DoS via deeply nested or expensive queries.

Cloud Spanner

TrueTimeGoogle's globally distributed atomic clock system. Provides bounded uncertainty intervals for wall-clock time. Used by Spanner to assign globally ordered commit timestamps for external consistency.
InterleavingPhysical co-location of child rows with parent rows in Spanner. Eliminates cross-split joins for hierarchical data. Defined via INTERLEAVE IN PARENT clause in DDL.
HotspotCondition where sequential primary keys cause all writes to land on a single split boundary, exhausting that split's capacity. Avoided with UUID or hash-distributed keys.
Stale ReadsSpanner reads at a past timestamp (e.g., 15 seconds ago). No quorum coordination needed — lower latency. Safe for non-critical reads. Configured via TimestampBound.
Mutations vs DMLMutations: buffer changes and apply atomically at commit. Faster, no query planning. DML: standard SQL INSERT/UPDATE, readable within transaction. Use DML when you need to read back changes in the same transaction.

Architecture Patterns

Outbox PatternReliability pattern: write event to outbox table in same transaction as business data. Background poller publishes to message broker. Prevents dual-write inconsistency.
Idempotency KeyClient-generated UUID attached to a mutating API request. Server caches response by key. Duplicate requests return cached response without re-executing. Critical for safe retries in distributed systems.
Optimistic LockingConcurrency control via version column. Update includes WHERE version = current — fails if concurrent update occurred. Caller retries. Best for low-contention scenarios.
CQRSCommand Query Responsibility Segregation. Separate models for reads (queries) and writes (commands). Read models can be denormalized for performance. Commands go through domain logic and emit events.
Saga PatternManages distributed transactions across microservices without 2PC. Choreography (events) or Orchestration (coordinator). Each step has a compensating transaction for rollback.

Security

Argon2idOWASP-recommended password hashing algorithm. Memory-hard (resists GPU/FPGA attacks). Three tuning parameters: time cost, memory, parallelism. Winner of Password Hashing Competition 2015.
bcryptAdaptive password hashing with adjustable cost factor. Industry standard since 1999. Auto-generates salt. Cost factor 12 ≈ 250ms on modern hardware. Spring Security's BCryptPasswordEncoder.
Crypto-EraseGDPR compliance technique. Replace PII with a UUID key; encrypt mapping in a separate table. Delete the key to make all references unresolvable. More practical than finding and deleting all PII copies.
CMEKCustomer-Managed Encryption Keys. GCP feature that lets you control encryption keys for Spanner, GCS, and other services via Cloud KMS. Required for compliance in some regulated industries.

Observability

SLI / SLO / SLASLI: measured metric (e.g., success rate). SLO: internal target (99.9%). SLA: external contract with consequences. Error budget = 100% - SLO. Drives reliability prioritisation decisions.
Four Golden SignalsGoogle SRE's core monitoring signals: Latency (response time), Traffic (req/s), Errors (failure rate), Saturation (resource utilization). Cover these before adding domain-specific metrics.
Distributed TracingPropagates a trace ID across service calls. Each operation creates a span with timing and metadata. Reconstructed as a trace tree in tools like Cloud Trace. Critical for diagnosing latency in microservices.
