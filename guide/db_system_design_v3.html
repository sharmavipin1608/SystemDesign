Golden rules

01
Exhaust Postgres before migrating. Most scaling problems are a missing index.
02
Redis cache solves most read problems in a day. A DB migration takes weeks.
03
Add a Kafka write buffer before sharding. It buys months of runway.
04
Never run analytics queries on your OLTP primary. They lock rows and kill perf.
05
Batch inserts: 100 rows in 1 INSERT is 10–50× faster than 100 single inserts.
06
Never make Elasticsearch your source of truth. It's an index, not a database.
Try these before sharding
1. Missing index
Run EXPLAIN ANALYZE. Most "scaling" issues are unindexed queries.
2. pgBouncer
100 connections → 10K virtual. Free throughput with zero schema change.
3. Read replicas
1 replica ≈ 2–4× read capacity. Route SELECTs there, writes to primary.
4. Partitioning
Declarative time partitions on a 2B-row table. Often obviates sharding.

Sharding techniques

Hash sharding
hash(shard_key) % N — uniform spread, no natural ordering.
PROS
Even load · No hotspots · Simple routing
CONS
Range queries scatter-gather · Resharding remaps ~all keys
Gotcha: resharding with mod-N remaps ~(N-1)/N of all keys. Use consistent hashing instead if you expect node changes.
Range sharding
Partition by value range — A–M on shard 1, N–Z on shard 2, or by date.
PROS
Range queries fast · Simple mental model
CONS
Hotspot on latest shard · Manual rebalancing
Gotcha: timestamp range sharding → all writes hit the latest shard. Fix: hash prefix + timestamp, or bucket by time + hash.
Consistent hashing
Nodes on a ring. Each key maps to nearest node clockwise. Virtual nodes even the load.
PROS
Minimal remapping on resize · Used by Cassandra, DynamoDB
CONS
Uneven without vnodes · Complex to implement
Gotcha: without virtual nodes one physical node can own 40%+ of the ring. Always configure vnodes (Cassandra default: 256).
Directory / lookup sharding
A lookup service maps each key to its shard explicitly. Full control over placement.
PROS
Full flexibility · Easy custom rebalancing
CONS
Lookup table = SPOF · Extra hop per request
Gotcha: if the lookup service goes down, nothing routes. Cache the lookup aggressively and replicate it — it's on the critical path.

Shard key selection

Good shard keys
High cardinality · Even distribution · No natural hotspot
user_id order_id UUID device_id
Bad shard keys
Low cardinality or monotonic = guaranteed hotspot
status country created_at boolean
Choosing the right key
Queries that include the shard key hit 1 shard. Without it — all shards queried in parallel (fan-out).
user_id for social tenant_id for SaaS geo for maps

Replicate vs shard vs both

Replicate when
Read traffic is high, writes are manageable
Data fits on a single node (<1TB)
Write QPS under node limit
Want simplicity first
Shard when
Write throughput exceeds single node
Data volume >1TB and growing fast
Read replicas alone aren't enough
Need horizontal write scale

Golden rules — sharding

01
Try declarative partitioning before sharding. Partitioning a 2B-row table by date is free and often enough.
02
Choosing the shard key is the most important decision. Get it wrong and you're stuck with hotspots forever.
03
Never choose a low-cardinality shard key. status, country, boolean — all writes for that value stack on one shard.
04
Cross-shard JOINs are 5–10× slower than single-shard. Design your access patterns around the shard key.
05
More than 5–7 read replicas adds WAL overhead on the primary and slows writes. Shard instead.
06
Resharding is painful. Over-provision shard count early — adding shards later remaps data across the cluster.

Golden rules — scaling

01
Cache hit rate is the multiplier. At 90% hit, 100K reads/s becomes 10K to the DB — 1 replica is fine.
02
Never exceed 5–7 read replicas per primary. Beyond that, WAL shipping overhead slows writes. Shard instead.
03
Design for the cache miss path, not the hit path. The miss path is what kills you under load.
04
Read-your-own-writes: route a user's reads to primary for 5–10s after their write. Prevents "my post disappeared" bugs from replica lag.
05
Add jitter to cache TTLs: TTL + random(0, 30s). Prevents cache stampede when hot keys expire simultaneously.
06
Monitor consumer lag, not just throughput. Lag tells you if your scaling strategy is actually keeping up.
