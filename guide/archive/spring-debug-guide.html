Chapter 01 · Foundations

Core Concepts & Bean Lifecycle

The Spring container, ApplicationContext, bean lifecycle, and the annotations that hold everything together.

ApplicationContext Bean Lifecycle @Component @Service @Repository @Configuration @Bean ComponentScan

Spring is fundamentally a container that manages object creation and wiring. If you understand what the container does — and what breaks when you step outside it — you can debug almost anything.

What Is the Spring Container?

The Spring container (ApplicationContext) is responsible for creating objects (beans), injecting their dependencies, and managing their lifecycle. When you write @Service on a class, you are telling the container: "Please manage this object for me."

🏭 Analogy

Think of Spring as a factory + warehouse. You submit blueprints (annotated classes). The factory produces exactly one copy of each object (singleton by default) and stores it in the warehouse. When any other object needs it, the factory delivers it — you never call new yourself. The moment you call new manually, Spring has no idea that object exists, and nothing gets injected into it.

Bean Lifecycle (Step by Step)

1
Container starts up
Spring scans the classpath for annotated classes (@Component, @Service, etc.) or reads @Bean methods in @Configuration classes.
2
Bean instantiation
Spring calls the constructor. At this point, @Autowired fields are not yet set. Never do real work in a constructor that depends on injected fields.
3
Dependency injection
Spring sets all @Autowired fields, constructor params, and setter params. Only after this step are dependencies available.
4
@PostConstruct / InitializingBean
If the bean has a method annotated @PostConstruct, it runs now. This is the right place for initialization that depends on injected fields.
5
Bean is ready for use
The bean enters the container and can be injected into other beans or used by the application.
6
@PreDestroy / shutdown
When the context closes, Spring calls @PreDestroy methods for cleanup.

The Core Stereotype Annotations

@Component
Generic Spring-managed bean. All other stereotypes extend this. Use when no other specific annotation fits.
@Service
Marks business logic classes. Semantically means "this is a service layer class." No extra behaviour over @Component currently, but communicates intent clearly.
@Repository
Marks data access classes. Unlike the others, it enables exception translation — raw SQL exceptions are wrapped into Spring's DataAccessException hierarchy.
@Controller
Marks web layer classes for Spring MVC. Methods return view names by default. Add @ResponseBody to return JSON/data instead.
@RestController
Shorthand for @Controller + @ResponseBody. All methods return the value directly as the HTTP response body — no view resolution.
@Configuration
Marks a class that contains @Bean methods. Spring enhances the class via CGLIB so that @Bean method calls are intercepted and return cached instances.

Component Scanning

Spring only discovers beans in packages it scans. By default in Spring Boot, it scans the package of the class annotated with @SpringBootApplication and all sub-packages.

🚨
Common Bug: Bean Not Found If you put a @Service class in a package outside the base package, Spring never finds it. You'll get NoSuchBeanDefinitionException. Fix: move the class into the scan path or add @ComponentScan("com.example.other").
01 Correct bean setup 02 Missing annotation 03 new vs Spring-managed 04 @Bean in non-@Configuration 05 @PostConstruct NPE trap
✓ CorrectWell-formed service bean with constructor injection
@Service
public class OrderService {

    private final OrderRepository orderRepository;

    // Constructor injection — preferred. Spring calls this.
    public OrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }

    public Order findById(Long id) {
        return orderRepository.findById(id).orElseThrow();
    }
}

✅ @Service tells Spring to manage this class. OrderRepository is injected via constructor — the field is final, meaning it can never be null after construction.

✗ BugMissing @Service — bean not in container
// No annotation — Spring doesn't know this class exists
public class PaymentService {

    @Autowired
    private PaymentRepository repo; // ← never injected

    public void processPayment() {
        repo.save(...); // NullPointerException at runtime
    }
}

❌ Without @Service (or @Component), Spring never creates this bean. Any class that tries to @Autowired PaymentService will fail with NoSuchBeanDefinitionException.

✗ BugUsing new keyword — Spring can't inject
@Service
public class CheckoutService {

    // WRONG: Creating directly — Spring has no control over this object
    private TaxService taxService = new TaxService();

    public BigDecimal calculateTotal(Cart cart) {
        // TaxService fields annotated @Autowired are all null here
        return taxService.calculate(cart);
    }
}

❌ When you call new TaxService(), you create a raw Java object — Spring never touches it. Any @Autowired fields inside TaxService will be null.

⚠ Edge Case@Bean method called directly — bypasses proxy
@Configuration
public class AppConfig {

    @Bean
    public DataSource dataSource() { return new HikariDataSource(...); }

    @Bean
    public JdbcTemplate jdbcTemplate() {
        // ✅ Correct: CGLIB proxy intercepts, returns singleton DataSource
        return new JdbcTemplate(dataSource());
    }
}

// What if @Configuration is missing?
public class AppConfig { // no @Configuration !
    @Bean
    public JdbcTemplate jdbcTemplate() {
        // ❌ dataSource() is a plain Java method call — creates a NEW DataSource
        return new JdbcTemplate(dataSource());
    }
}

⚠ Without @Configuration, @Bean methods behave as normal Java methods. Each call creates a new instance — you lose singleton semantics.

@PostConstructNPE trap: using @Autowired field in constructor
@Service
public class CacheService {

    @Autowired
    private UserRepository userRepo;

    // ❌ WRONG: userRepo is null here — injection hasn't happened yet
    public CacheService() {
        userRepo.findAll(); // NullPointerException
    }

    // ✅ CORRECT: runs AFTER all @Autowired fields are set
    @PostConstruct
    public void init() {
        userRepo.findAll(); // safe — userRepo is injected
    }
}

✅ @PostConstruct is the correct hook for initialization logic that depends on injected beans.

🧠 Quiz — Chapter 1
Q1. A class annotated with @Service has an @Autowired field that is always null at runtime. What is the most likely cause?
  • A) The field type has no default value
  • B) The object was instantiated with new instead of being injected by Spring
  • C) @Service is only for method-level use
  • D) Spring doesn't support field injection
✅ Answer: B. When you use new MyService(), Spring never processes the object. All @Autowired fields remain null. Always obtain Spring beans via injection, not direct construction.
Q2. You move a @Repository class from com.app.data to com.util.data. Suddenly it's not found. Why?
  • A) @Repository doesn't work outside com.app
  • B) The new package is outside the component scan base package
  • C) You need to restart the JVM
  • D) @Repository only works in web applications
✅ Answer: B. Spring Boot scans from the @SpringBootApplication package downward. Moving a class outside that tree means Spring never discovers it. Fix: add @ComponentScan("com.util") or move the class back.
💡
Key Takeaway The Spring container only knows about objects it created. The moment you use new, you step outside the container — no injection, no AOP, no transactions. This single rule explains 80% of Spring bugs.
Chapter 02 · Critical Topic ⚠

@Transactional Deep Dive

The most tested annotation in Spring interviews. Understand the proxy, propagation, rollback rules, and all the silent failure modes.

AOP Proxy Propagation Isolation rollbackFor Self-Invocation Checked Exceptions Private Methods readOnly

@Transactional is powered by a proxy. Every rule — why self-invocation fails, why private methods don't work, why checked exceptions don't rollback — follows from this one fact.

How @Transactional Works Under the Hood

🔀 Analogy — The Proxy Bouncer

Imagine your OrderService is a VIP room. Spring wraps it in a proxy bouncer that stands at the door. When external code calls orderService.placeOrder(), the bouncer intercepts the call, opens a transaction, lets the call through, and then commits or rolls back based on outcome. But if placeOrder() calls this.validateOrder() directly inside the room — the bouncer never sees it. Self-calls bypass the proxy entirely.

1
Spring creates a proxy around your bean
At startup, if any method has @Transactional, Spring wraps the bean in a CGLIB subclass or JDK proxy that intercepts method calls.
2
External call hits the proxy
The proxy checks: does this method have @Transactional? If yes, it begins a transaction, then delegates to the real method.
3
Method executes
The real bean method runs inside the transaction context.
4
Proxy decides: commit or rollback
If the method returns normally → commit. If it throws an unchecked exception (RuntimeException/Error) → rollback. Checked exceptions → commit by default (dangerous!).

Propagation Types — The Most Tested

Rollback Rules — The Tricky Default

🚨
Default: Only RuntimeException and Error trigger rollback. If your method throws a checked exception (e.g. IOException, SQLException), Spring will commit the transaction by default! To rollback on checked exceptions: @Transactional(rollbackFor = Exception.class)
rollbackFor vs noRollbackFor
@Transactional(rollbackFor = MyCheckedException.class) — rollback on this checked exception too.
@Transactional(noRollbackFor = ItemNotFoundException.class) — do NOT rollback even though it's a RuntimeException.

Silent Failure Modes — The Big 4

❌ Self-Invocation
Calling a @Transactional method from within the same class using this.method(). The call goes directly to the object, bypassing the proxy. No transaction is created.
❌ Private Method
@Transactional on a private method is silently ignored. The proxy cannot override private methods. Spring won't warn you — the annotation is simply ineffective.
❌ final Class/Method
CGLIB creates a subclass of your bean. If the class or method is final, CGLIB can't override it. The transaction annotation is ignored. No warning.
❌ Checked Exception
Throwing IOException from a @Transactional method. Spring commits the transaction — your data changes are persisted even though an exception was thrown.

readOnly = true

Setting @Transactional(readOnly = true) is a performance hint. The database driver and JPA can skip dirty checking, flush operations, and lock acquisition. Use it for all query-only methods.

⚠️
readOnly does not enforce read-only. It's a hint, not a constraint. You can still accidentally modify data — but the changes may or may not be flushed to the DB depending on the JPA provider.

✓ CorrectProper @Transactional on a public service method
@Service
public class OrderService {

    @Transactional
    public void placeOrder(Order order) {
        orderRepo.save(order);
        inventoryService.deduct(order);
        // If any RuntimeException is thrown → both are rolled back
    }

    @Transactional(readOnly = true)
    public List<Order> getOrders(Long userId) {
        return orderRepo.findByUserId(userId);
    }
}

✅ Public method, annotated directly on service. readOnly for queries. Any RuntimeException in placeOrder rolls back both saves atomically.

✗ BugSelf-invocation — proxy bypassed
@Service
public class NotificationService {

    @Transactional
    public void sendBulk(List<User> users) {
        for (User u : users) {
            // ❌ Calling this.sendOne() — goes directly to the real object
            // The @Transactional on sendOne() is IGNORED
            sendOne(u);
        }
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void sendOne(User user) {
        // Intended: each sendOne gets its own transaction
        // Actual: runs in the same transaction as sendBulk (or no tx)
        repo.save(new Notification(user));
    }
}

// FIX: Inject a self-reference or extract sendOne to a separate bean
@Service
public class NotificationService {
    @Autowired private NotificationService self; // Spring injects the proxy

    public void sendBulk(List<User> users) {
        for (User u : users) self.sendOne(u); // ✅ goes through proxy
    }
}

❌ this.sendOne() bypasses the Spring proxy. The REQUIRES_NEW propagation is never applied. The fix is to call through the proxy by injecting a self-reference.

✗ Bug@Transactional on private method — silently ignored
@Service
public class PaymentService {

    public void processPayment(Payment p) {
        doProcess(p); // calls the private method below
    }

    // ❌ @Transactional on private method — proxy cannot override this
    @Transactional
    private void doProcess(Payment p) {
        paymentRepo.save(p);
        ledgerRepo.record(p);
        // No transaction wraps this — if ledgerRepo.record() throws,
        // paymentRepo.save() is NOT rolled back
    }
}

❌ Spring generates no warning. The annotation is silently discarded. Always put @Transactional on public methods.

⚠ TrickyChecked exception does NOT rollback by default
@Service
public class FileService {

    // ❌ IOException is a checked exception
    // Spring will COMMIT even if IOException is thrown
    @Transactional
    public void importData(File file) throws IOException {
        dataRepo.save(parseFile(file)); // saved to DB
        fileStorage.move(file);         // throws IOException
        // dataRepo.save IS committed despite the exception!
    }

    // ✅ Fix: explicitly rollback on checked exceptions
    @Transactional(rollbackFor = Exception.class)
    public void importDataFixed(File file) throws IOException {
        dataRepo.save(parseFile(file));
        fileStorage.move(file); // now this WILL trigger rollback
    }
}

⚠ This is a frequent interview question. Default rollback triggers only on RuntimeException and Error. For checked exceptions, always add rollbackFor.

PatternREQUIRES_NEW for independent audit logging
@Service
public class AuditService {

    // REQUIRES_NEW: always commits independently of outer transaction
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void log(String event) {
        auditRepo.save(new AuditEntry(event));
    }
}

@Service
public class OrderService {
    @Autowired AuditService auditService;

    @Transactional
    public void placeOrder(Order order) {
        orderRepo.save(order);
        auditService.log("ORDER_PLACED"); // commits in its own tx
        throw new RuntimeException("Something went wrong");
        // orderRepo.save IS rolled back
        // auditService.log IS committed (separate tx already committed)
    }
}

✅ Audit log is committed even when the outer transaction rolls back. This is the canonical use case for REQUIRES_NEW.

PatternClass-level vs method-level @Transactional
// Class-level: applies to ALL public methods as default
@Service
@Transactional(readOnly = true)
public class ProductService {

    public Product findById(Long id) {
        return repo.findById(id).orElseThrow(); // inherits readOnly=true
    }

    // Method-level overrides class-level
    @Transactional  // readOnly=false (default)
    public void updatePrice(Long id, BigDecimal price) {
        Product p = repo.findById(id).orElseThrow();
        p.setPrice(price);
        repo.save(p); // write-capable transaction
    }
}

✅ Class-level annotation is the default for all methods. Individual methods can override with their own @Transactional. Useful pattern: class-level readOnly=true, override on write methods.

🧠 Quiz — Chapter 2
Q1. A @Transactional method throws an IOException. What happens to the transaction?
  • A) Transaction is rolled back automatically
  • B) Transaction is committed — IOException is checked, not rolled back by default
  • C) Transaction is suspended
  • D) Spring converts it to a RuntimeException
✅ Answer: B. Spring only rolls back on RuntimeException and Error by default. IOException is a checked exception — the transaction commits. Use rollbackFor = Exception.class to change this.
Q2. Method A (no @Transactional) calls Method B (REQUIRES_NEW) in the same class. What happens to Method B's transaction?
  • A) A new transaction is created for Method B
  • B) Method B runs without a transaction
  • C) Method B shares Method A's transaction
  • D) A transaction exception is thrown
✅ Answer: B. Since A calls B as this.B(), it bypasses the proxy entirely. The REQUIRES_NEW annotation is never processed. B runs with no transaction at all — exactly as if there were no annotation.
Q3. You annotate a private method with @Transactional. What happens?
  • A) Spring throws a startup error
  • B) Spring warns you in the logs
  • C) The annotation is silently ignored — no transaction is created
  • D) A transaction is created but can't be committed
✅ Answer: C. Spring silently ignores @Transactional on private methods. No error, no warning. The proxy cannot override private methods, so the advice is never applied.
🔑
The One Rule for @Transactional It works via a proxy that wraps your bean. Any scenario that bypasses the proxy (self-invocation, private methods, final classes, calling new MyService()) silently disables the transaction. Always verify: is this call going through the proxy?
Chapter 03 · Web Layer

Spring MVC

Controllers, request mapping, input binding, and the annotations that decide what your endpoints return.

@RestController @RequestMapping @RequestBody @PathVariable @RequestParam @ResponseStatus @ExceptionHandler @ControllerAdvice

The web layer is where inputs enter and outputs leave your application. Wrong annotations mean wrong HTTP codes, null parameters, or views instead of JSON — all classic interview bugs.

@Controller vs @RestController

📺 Analogy

@Controller is a TV director — it returns a script name (view name) and the template engine renders the actual show. @RestController is a live broadcast — it returns the content itself directly to the viewer with no template engine involved.

@Controller
Return value is treated as a view name resolved by a template engine (Thymeleaf, JSP). To return data directly, you must add @ResponseBody on the method or class.
@RestController
= @Controller + @ResponseBody. Every method's return value is serialized (usually to JSON) and written directly into the HTTP response body.

HTTP Status Codes

By default, a successful controller method returns 200 OK. Use @ResponseStatus to override, or return a ResponseEntity with explicit status.

@ResponseStatus
On a method: always returns that status.
On an exception class: when that exception is thrown, return that status.
@ResponseStatus(HttpStatus.CREATED)

return ResponseEntity.status(201).body(created);

🚨
Common Bug: Missing @ResponseBody If you use @Controller but forget @ResponseBody, the method returns a string that Spring treats as a view name. You'll see a "Whitelabel Error Page" saying view not found, or a TemplateInputException.
01 Correct REST endpoint 02 Missing @ResponseBody 03 Wrong input annotation 04 Global exception handler 05 HTTP status control
✓ CorrectWell-formed REST controller
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    @GetMapping("/{id}")
    public OrderDto getOrder(@PathVariable Long id) {
        return orderService.findById(id); // serialized to JSON → 200 OK
    }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public OrderDto createOrder(@RequestBody CreateOrderRequest req) {
        return orderService.create(req); // 201 Created
    }

    @GetMapping
    public List<OrderDto> listOrders(
        @RequestParam(defaultValue = "OPEN") String status) {
        return orderService.findByStatus(status);
    }
}

✅ @RestController handles JSON serialization. @PathVariable from URL, @RequestBody from POST body, @RequestParam from query string.

✗ Bug@Controller without @ResponseBody — returns view name
@Controller  // ← NOT @RestController
public class UserController {

    @GetMapping("/user/{id}")
    public UserDto getUser(@PathVariable Long id) {
        // ❌ Spring tries to resolve "UserDto@123abc" as a view template name
        // Result: TemplateInputException — no template "UserDto@123abc" found
        return userService.findById(id);
    }

    // ✅ Fix Option 1: Add @ResponseBody
    @GetMapping("/user/{id}")
    @ResponseBody
    public UserDto getUser(@PathVariable Long id) { ... }

    // ✅ Fix Option 2: Change to @RestController
}

❌ Without @ResponseBody, Spring MVC treats the return value as a view name and tries to render a template. The fix is to either add @ResponseBody or switch to @RestController.

✗ BugWrong input binding annotation
@RestController
public class ProductController {

    // URL: POST /products/activate?productId=5
    @PostMapping("/products/activate")
    public void activate(@PathVariable Long productId) {
        // ❌ @PathVariable looks in the URL path {productId}
        // But productId is a query param, not a path segment
        // Result: MissingPathVariableException
    }

    // ✅ Fix
    @PostMapping("/products/activate")
    public void activate(@RequestParam Long productId) { ... }

    // Also wrong: using @RequestParam for a JSON body
    @PostMapping("/products")
    public void create(@RequestParam ProductDto dto) {
        // ❌ @RequestParam reads from query string, not body
        // Fix: use @RequestBody
    }
}

❌ The most common annotation confusion: @PathVariable = URL segment, @RequestParam = query string, @RequestBody = JSON request body. Mixing them gives nulls or exceptions.

PatternGlobal exception handler with @ControllerAdvice
@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(EntityNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    @ResponseBody
    public ErrorResponse handleNotFound(EntityNotFoundException ex) {
        return new ErrorResponse("NOT_FOUND", ex.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    @ResponseBody
    public ErrorResponse handleValidation(MethodArgumentNotValidException ex) {
        String msg = ex.getBindingResult().getFieldErrors().stream()
            .map(FieldError::getDefaultMessage).collect(joining(", "));
        return new ErrorResponse("VALIDATION_ERROR", msg);
    }
}

✅ @ControllerAdvice applies globally across all controllers. Centralize all error mapping here rather than duplicating @ExceptionHandler in every controller.

Input/OutputWhat HTTP status does each endpoint return?
@RestController
public class QuizController {

    @GetMapping("/a")
    public String methodA() { return "hello"; }
    // Output: 200 OK, body: hello

    @PostMapping("/b")
    @ResponseStatus(HttpStatus.CREATED)
    public void methodB() {}
    // Output: 201 Created, empty body

    @DeleteMapping("/c")
    public ResponseEntity<Void> methodC() {
        return ResponseEntity.noContent().build();
    }
    // Output: 204 No Content

    @GetMapping("/d")
    public String methodD() { throw new RuntimeException("fail"); }
    // Output: 500 Internal Server Error (no handler configured)
}

✅ Classic input/output question: know the default (200), @ResponseStatus overrides, ResponseEntity for programmatic control, and unhandled exceptions → 500.

🧠 Quiz — Chapter 3
Q1. A @Controller method returns a UserDto object without any additional annotation. What does the client receive?
  • A) JSON representation of UserDto with 200 OK
  • B) A template rendering error or 500 status
  • C) The toString() of UserDto
  • D) Empty 200 response
✅ Answer: B. @Controller without @ResponseBody treats return values as view names. Spring will try to find a template with the name of the object's toString() and fail with a TemplateInputException or 500.
Q2. URL: GET /products?category=electronics. Which annotation correctly binds category?
  • A) @PathVariable String category
  • B) @RequestBody String category
  • C) @RequestParam String category
  • D) @RequestHeader String category

Key Takeaway Remember the three input annotations: @PathVariable (URL segment), @RequestParam (query string), @RequestBody (JSON body). And remember: @Controller needs @ResponseBody to return data — @RestController includes it automatically.
Chapter 04 · DI Bugs

Dependency Injection Bugs

NoSuchBeanDefinitionException, NoUniqueBeanDefinitionException, circular dependencies, and scope mismatches.

@Autowired @Qualifier @Primary @Lazy Circular Dependency Scope Mismatch NoSuchBeanDefinition NoUniqueBeanDefinition

Most DI bugs produce a clear exception message — if you know what to look for. Learn to read the error, locate the cause, and apply the right annotation fix.

The Three DI Injection Styles

Field Injection ⚠
@Autowired private UserRepo repo;
Convenient but hides dependencies. Makes testing harder (can't inject mocks without reflection). Not recommended for production code.
Constructor Injection ✅
public Service(UserRepo repo)
Preferred. Dependencies are explicit. Field can be final. Works with standard unit testing without Spring. Detects circular deps at startup.
Setter Injection
@Autowired public void setRepo(UserRepo r)
Used when the dependency is truly optional. Less common. Cannot make the field final.

Common DI Exceptions

ExceptionCauseFix
NoSuchBeanDefinitionExceptionSpring can't find a bean of the requested typeAdd missing annotation, fix scan path, add @Bean
NoUniqueBeanDefinitionExceptionMultiple beans of the same type, Spring doesn't know which to injectAdd @Qualifier or mark one @Primary
BeanCurrentlyInCreationExceptionCircular dependency (A needs B, B needs A)Redesign, use setter injection, or @Lazy
UnsatisfiedDependencyExceptionWrapper for the above — check the cause messageFind root cause in the exception chain

@Qualifier and @Primary

When two beans implement the same interface, Spring doesn't know which to inject. Use @Primary to designate the default, or @Qualifier("name") to pick a specific one.

Scope Mismatch Bug

🚨
Injecting a short-lived bean into a long-lived bean — e.g. injecting a @RequestScope bean into a @Singleton. The singleton is created once and holds a stale reference. Fix: use a scoped proxy: @Scope(value="request", proxyMode=ScopedProxyMode.TARGET_CLASS)

@Lazy

By default, Spring creates all singleton beans at startup. @Lazy on a bean or injection point defers creation until first use. Useful for expensive beans or breaking circular dependency deadlocks at startup.

✓ CorrectConstructor injection — the preferred style
@Service
public class ReportService {

    private final ReportRepository repo;
    private final EmailService emailService;

    // No @Autowired needed with single constructor in Spring 4.3+
    public ReportService(ReportRepository repo, EmailService emailService) {
        this.repo = repo;
        this.emailService = emailService;
    }

✅ Dependencies are explicit, final, and guaranteed non-null. Easy to unit test by passing mocks to the constructor directly.

✗ BugTwo beans, same type — NoUniqueBeanDefinitionException
public interface NotificationSender { void send(String msg); }

@Service
public class EmailSender implements NotificationSender { ... }

@Service
public class SmsSender implements NotificationSender { ... }

@Service
public class AlertService {
    @Autowired
    private NotificationSender sender; // ❌ Which one? Spring doesn't know

}

@Service @Primary
public class EmailSender implements NotificationSender { ... }

// Fix 2: @Qualifier to pick specific one
@Autowired
@Qualifier("smsSender")

❌ Two beans implement the same interface. Spring can't decide. Use @Primary for the default choice, or @Qualifier to be explicit.

public class ServiceA {

public class ServiceB {
    private final ServiceA a;

}
// ❌ BeanCurrentlyInCreationException at startup
// A needs B, but B needs A, which needs B... infinite loop

// Fix 1: Redesign (best) — introduce a third service
// Fix 2: @Lazy on one injection point
@Service
public class ServiceA {
    public ServiceA(@Lazy ServiceB b) { this.b = b; }
    // ServiceB is created lazily on first use, breaking the startup cycle
}

❌ Spring detects circular constructor injection at startup. @Lazy defers one bean's creation, breaking the cycle. But it's usually a design smell — consider refactoring.

⚠ Scope MismatchRequest-scoped bean injected into singleton
@Component
@Scope("request")  // new instance per HTTP request
public class RequestContext {
    private String userId;
}

@Service  // singleton — created once
public class UserService {
    @Autowired
    private RequestContext ctx; // ❌ This is the instance from startup time
    // After first request, ctx is stale — it always has the first request's data
}

// ✅ Fix: use a scoped proxy
@Component
@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class RequestContext {
    // Now Spring injects a proxy that forwards each call to the current request's instance
}

⚠ Without a scoped proxy, the singleton holds a reference to a stale request-scoped object. The proxy ensures each method call is forwarded to the correct per-request instance.

PatternOptional dependency with @Autowired required=false
@Service
public class NotificationService {

    // ❌ Without required=false, startup fails if no SlackClient bean exists
    @Autowired
    private SlackClient slackClient;

    // ✅ Optional dependency — null if bean not present
    @Autowired(required = false)
    private SlackClient slackClient;

    // Even better: use Java Optional
    @Autowired
    private Optional<SlackClient> slackClient;

    public void notify(String msg) {
        slackClient.ifPresent(c -> c.send(msg));
    }
}

✅ Use required=false or Optional<T> for beans that may or may not be available depending on configuration (e.g., integration clients only active in prod).

LazyInitializationException — The Most Common JPA Bug

📦 Analogy

JPA lazy loading is like ordering furniture with "assemble on delivery." The box arrives (the entity), but the furniture inside (related entities) is only unpacked when you open it. If the delivery person (transaction/session) has left already, and you try to open the box — you get a LazyInitializationException: the session is closed.

🚨
LazyInitializationException: could not initialize proxy — no Session This means you tried to access a lazy-loaded collection or association after the transaction/session was closed. Fixes: load within the transaction, use EAGER fetch, use @Transactional on the calling service, or use a JPQL JOIN FETCH.

N+1 Query Problem

Loading a list of Orders, then for each Order accessing its lazy-loaded Customer — this fires 1 query for Orders + N queries for Customers. Fix with JOIN FETCH in JPQL or @EntityGraph.

FetchType

FetchTypeDefault forBehaviour
LAZY@OneToMany, @ManyToManyRelated entities loaded only when accessed. Efficient but risky outside transaction.
EAGER@ManyToOne, @OneToOneRelated entity loaded immediately with the parent. Safer but can cause performance issues.
01 Correct entity + repo 02 LazyInitializationException 03 N+1 problem + fix 04 Custom @Query 05 Missing @Repository exception translation
✓ CorrectEntity with JpaRepository
@Entity
@Table(name = "orders")
public class Order {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String status;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "customer_id")
    private Customer customer;

    protected Order() {} // Required no-arg constructor for JPA
}

@Repository
public interface OrderRepository extends JpaRepository<Order, Long> {
    List<Order> findByStatus(String status);
    Optional<Order> findByIdAndCustomerId(Long id, Long customerId);
}

✅ Entity has a no-arg constructor. Repository extends JpaRepository. Derived query methods are auto-implemented by Spring Data from the method name.

✗ BugLazyInitializationException — accessing lazy relation outside transaction
@Service
public class OrderService {

    public String getCustomerName(Long orderId) {
        Order order = orderRepo.findById(orderId).orElseThrow();
        // Transaction closed after findById returns
        // ❌ Accessing lazy-loaded Customer NOW — session is gone
        return order.getCustomer().getName(); // LazyInitializationException!
    }

    // ✅ Fix 1: Add @Transactional to keep session open
    @Transactional
    public String getCustomerName(Long orderId) {
        Order order = orderRepo.findById(orderId).orElseThrow();
        return order.getCustomer().getName(); // ✅ session still open
    }

    // ✅ Fix 2: JOIN FETCH to load eagerly in one query
}

❌ Spring Data repositories open a transaction per method call. When findById returns, the session closes. Accessing lazy fields after that throws. Fix: keep a transaction open via @Transactional.

⚠ PerformanceN+1 problem and JOIN FETCH fix
// ❌ N+1: 1 query for orders + N queries for each customer
@Transactional
public List<String> getAllCustomerNames() {
    return orderRepo.findAll().stream()
        .map(o -> o.getCustomer().getName()) // fires 1 query per order
        .collect(toList());
}

// ✅ Fix: JOIN FETCH loads customer in the same query
public interface OrderRepository extends JpaRepository<Order, Long> {

    @Query("SELECT o FROM Order o JOIN FETCH o.customer")
    List<Order> findAllWithCustomer();
}

⚠ JOIN FETCH tells JPQL to load the association in a single SQL JOIN query, eliminating the N+1 problem. This is one of the most commonly asked JPA performance questions.

PatternCustom @Query — JPQL and native SQL
public interface ProductRepository extends JpaRepository<Product, Long> {

    // JPQL — uses entity/field names, not table/column names
    @Query("SELECT p FROM Product p WHERE p.price < :maxPrice AND p.category = :cat")
    List<Product> findCheapByCategory(@Param("maxPrice") BigDecimal max,
                                       @Param("cat") String category);

    // Native SQL — use actual table/column names
    @Query(value = "SELECT * FROM products WHERE price < ?1", nativeQuery = true)
    List<Product> findCheapNative(BigDecimal maxPrice);

    // Modifying query — must have @Modifying + @Transactional
    @Modifying
    @Transactional
    @Query("UPDATE Product p SET p.stock = 0 WHERE p.id = :id")
    int clearStock(@Param("id") Long id);
}

✅ JPQL queries use entity class and field names. Native queries use real SQL. @Modifying is required for UPDATE/DELETE queries — and they also need @Transactional.

Pattern@Repository exception translation
// Without @Repository — raw JDBC/JPA exceptions propagate
public class RawUserRepo {
    public void save(User u) {
        // throws org.hibernate.exception.ConstraintViolationException
        // caller must catch Hibernate-specific exceptions — tight coupling
    }
}

// With @Repository — Spring translates to DataAccessException hierarchy
@Repository
public class UserRepo {
    public void save(User u) {
        // throws DataIntegrityViolationException (Spring)
        // Spring DataAccessException — database-independent, easier to catch
    }
}

// Caller can catch Spring's exception without knowing about Hibernate:
try {
    userRepo.save(user);
} catch (DataIntegrityViolationException e) {
    // handle duplicate key, constraint violation etc.
}

✅ @Repository enables Spring's PersistenceExceptionTranslationPostProcessor. Raw Hibernate/JPA exceptions are translated into Spring's database-agnostic DataAccessException hierarchy.

🧠 Quiz — Chapter 5
Q1. You load an Order with a lazy-loaded Customer. You access order.getCustomer().getName() outside any transaction. What happens?
  • A) Returns null silently
  • B) LazyInitializationException — could not initialize proxy, no Session
  • C) Spring opens a new session automatically
  • D) EntityNotFoundException
✅ Answer: B. Once the transaction/session closes, lazy associations cannot be loaded. Accessing them throws LazyInitializationException. Fix: add @Transactional to the calling method or use JOIN FETCH.
Q2. What is the N+1 problem?
  • A) An entity has N+1 fields instead of N
  • B) Loading N entities, then firing 1 extra query per entity to load a related association, totaling N+1 queries
  • C) Having more than N+1 indexes on a table
  • D) A JPA constraint violation
✅ Answer: B. N+1 means: 1 query to load the list + N queries to load the lazy association for each item. For 100 orders, that's 101 SQL queries. Fix with JOIN FETCH to load everything in 1 query.
💡
Key Takeaway Two things to always remember in JPA: (1) lazy-loaded associations require an open session — always access them inside a @Transactional method; (2) loading collections in a loop causes N+1 queries — use JOIN FETCH or EntityGraph.

@Value

Chapter 07 · AOP

AOP Basics

The proxy model that powers @Transactional, @Cacheable, and @Async — and why final classes break them all.

CGLIB Proxy JDK Dynamic Proxy @Aspect @Around Pointcut JoinPoint Advice

You don't need to write AOP aspects in most jobs — but you need to understand that Spring's most important annotations are all AOP-based, and the proxy model is why they have specific limitations.

Two Proxy Types

JDK Dynamic Proxy
Used when your bean implements an interface. Spring creates a proxy that implements the same interface and delegates method calls to the real object. Only methods declared in the interface are intercepted.
CGLIB Proxy
Used when your bean has no interface (or Spring Boot default). Spring creates a subclass of your bean class at runtime. Consequence: the class and method must not be final — a final class cannot be subclassed.
🚨
final class + @Transactional = silent failure. CGLIB cannot subclass a final class. The proxy is not created, the annotation is ignored, and there is no error. Same for Kotlin classes, which are final by default — must add open or use the Kotlin Spring plugin.

AOP Terminology (Read-Level)

TermMeaningExample
AspectThe cross-cutting concern module (the class)@Aspect class LoggingAspect
AdviceThe code to run — what, when@Before, @After, @Around
PointcutExpression that matches which methods the advice applies toexecution(* com.app.service.*.*(..))
JoinPointThe actual method invocation being interceptedAvailable as parameter in advice methods
WeavingApplying aspects to target classesSpring does this at startup (load-time)

Advice Types

@Before
Runs before the method. Cannot prevent method execution (use @Around for that).
@After
Runs after the method, regardless of outcome (like finally).
@AfterReturning
Runs after successful return. Has access to the return value.
@AfterThrowing
Runs only if the method throws. Has access to the exception.
@Around
Full control — wraps the method. Must call proceed() to execute the original. Used by @Transactional internally.

✓ PatternLogging aspect with @Before and @AfterReturning
@Aspect
@Component
public class LoggingAspect {

    // Pointcut: match all methods in service package
    @Pointcut("execution(* com.app.service.*.*(..))")
    public void serviceMethods() {}

    @Before("serviceMethods()")
    public void logBefore(JoinPoint jp) {
        System.out.println("Calling: " + jp.getSignature().getName());
    }

    @AfterReturning(pointcut = "serviceMethods()", returning = "result")
    public void logAfter(JoinPoint jp, Object result) {
        System.out.println("Returned: " + result);
    }
}

✅ The aspect class needs both @Aspect (tells Spring this is an AOP class) and @Component (makes it a Spring-managed bean so it gets picked up).

Deep DiveHow @Transactional is implemented — @Around advice
// This is conceptually what Spring's transaction advisor does internally
@Around("@annotation(org.springframework.transaction.annotation.Transactional)")
public Object manageTransaction(ProceedingJoinPoint pjp) throws Throwable {
    TransactionStatus tx = txManager.getTransaction(txDef);
    try {
        Object result = pjp.proceed(); // call the actual method
        txManager.commit(tx);
        return result;
    } catch (RuntimeException e) {
        txManager.rollback(tx);  // rollback on unchecked
        throw e;
    } catch (Exception e) {
        txManager.commit(tx);    // commit on checked by default!
        throw e;
    }
}
// This explains EVERY @Transactional behavior you've learned

✅ Understanding this pattern makes every @Transactional rule obvious: it's just an @Around advice that begins/commits/rolls back a transaction. Self-invocation bypasses this advice entirely.

  • C) The method should be static

Key Takeaway Spring AOP works by creating a proxy subclass or interface implementation. Any restriction that prevents proxying — final, private, this. calls — silently disables the aspect. All of @Transactional, @Cacheable, @Async, and custom aspects share this limitation.

Spot the BugFind all bugs in this code snippet
@Controller                              // Bug 1
public class UserController {

    private UserService userService = new UserService(); // Bug 2

    @GetMapping("/user/{id}")
    public UserDto getUser(@RequestParam Long id) { // Bug 3
        return userService.findById(id);
    }
}

public class UserService {              // Bug 4
    @Autowired
    private UserRepository repo;

    @Transactional
    private void deleteUser(Long id) {    // Bug 5
        repo.deleteById(id);
    }
}

Bug 1: @Controller without @ResponseBody — getUser() will try to resolve UserDto as a view name.
Bug 2: new UserService() — Spring doesn't manage this object. UserRepository inside will be null.
Bug 3: @RequestParam but the URL has {id} as a path variable — should be @PathVariable.
Bug 4: Missing @Service — Spring won't create a bean for this class.
Bug 5: @Transactional on a private method — silently ignored.

TransactionalDoes the transaction rollback? Trace the scenarios
@Service
public class PaymentService {

    @Transactional
    public void scenario1() {
        repo.save(entity);
        throw new RuntimeException("runtime");
    } // → ROLLBACK (RuntimeException triggers rollback)

    @Transactional
    public void scenario2() throws IOException {
        repo.save(entity);
        throw new IOException("checked");
    } // → COMMIT (IOException is checked — no rollback by default!)

    @Transactional(rollbackFor = Exception.class)
    public void scenario3() throws IOException {
        repo.save(entity);
        throw new IOException("checked");
    } // → ROLLBACK (rollbackFor overrides default)

    @Transactional
    public void scenario4() {
        innerMethod(); // self-invocation
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void innerMethod() {
        repo.save(entity);
        throw new RuntimeException();
    } // → What happens? innerMethod runs in scenario4's tx (self-invocation bypass)
    // → scenario4's tx is ROLLED BACK (RuntimeException from inner call)
    // → REQUIRES_NEW never applied — no second tx created
}

Trace each scenario until you can answer immediately without thinking. Scenario 4 is the hardest — self-invocation means REQUIRES_NEW is ignored and the exception propagates into the outer transaction.

Top Interview Questions with Answers

🎯 Final Assessment — Mix of All Topics
Q1. Service A (no @Transactional) calls Service B (REQUIRES_NEW) externally (different beans). What happens?
  • A) B runs without any transaction
  • B) B creates its own new transaction, independent of A
  • C) B joins A's transaction
  • D) Exception is thrown
✅ Answer: B. Different beans = call goes through proxy = REQUIRES_NEW is applied. B creates its own transaction. This is the correct behaviour because the call goes through the Spring proxy on Service B.
Q2. What annotation would you add to allow a Singleton bean to receive a fresh request-scoped bean on every HTTP request?
  • A) @Primary on the request-scoped bean
  • B) @Scope(value="request", proxyMode=ScopedProxyMode.TARGET_CLASS)
  • C) @Lazy on the injection point
  • D) @Qualifier("request")
✅ Answer: B. A scoped proxy wraps the request-scoped bean. The singleton holds the proxy, which forwards each method call to the current request's instance of the bean.
Q3. A @Transactional method successfully inserts row A, then throws a NullPointerException. What is the final state of the database?
  • A) Row A is committed
  • B) Row A is rolled back — NullPointerException is a RuntimeException
  • C) The behaviour is undefined
  • D) Row A is committed because the insert succeeded before the exception
✅ Answer: B. NullPointerException extends RuntimeException. @Transactional rolls back on RuntimeException by default. The entire transaction is rolled back — row A is NOT in the database.
Q4. What is the difference between @Component and @Bean?
  • A) @Bean is for services, @Component is for utilities
  • B) @Component is on a class Spring can instantiate; @Bean is a factory method for objects you instantiate yourself (e.g. third-party classes)
  • C) They are identical
  • D) @Component supports injection; @Bean does not
✅ Answer: B. @Component annotates your own class — Spring calls its constructor. @Bean is a method in a @Configuration class where you write the construction code yourself. Use @Bean for third-party classes you can't annotate (like ObjectMapper, RestTemplate).
Q5. Why does @Transactional silently fail on a private method?
  • A) Spring doesn't scan private methods
  • B) The AOP proxy cannot override private methods, so the advice never runs
  • C) @Transactional requires public visibility by contract
  • D) It's a JVM limitation
✅ Answer: B. @Transactional works via AOP proxy (CGLIB subclass). A subclass cannot override private methods. The proxy exists but can't intercept the call — the advice (transaction management) is never applied. No error is thrown.

Key Rules Cheat Sheet

@Transactional works only when: (1) called from outside the class through the proxy, (2) on a public method, (3) class is not final. Any violation = silent no-op.
Rollback rules: RuntimeException/Error → rollback. Checked Exception → commit (unless rollbackFor specified).
DI exceptions: NoSuchBean = missing annotation or wrong package. NoUniqueBean = two beans same type, use @Qualifier or @Primary. BeanCurrentlyInCreation = circular dep, use @Lazy or redesign.
Web layer: @PathVariable = URL segment. @RequestParam = query string. @RequestBody = JSON body. @Controller without @ResponseBody returns view name — add @ResponseBody or use @RestController.
JPA: Access lazy relations inside @Transactional. Loop + lazy = N+1 queries → use JOIN FETCH. @Modifying required for UPDATE/DELETE @Query.
Silent failures: @Transactional on private method. @Cacheable without @EnableCaching. @Async without @EnableAsync. @Scheduled without @EnableScheduling. @Bean in non-@Configuration class — no singleton guarantee.
