Sharding techniques

Hash sharding
hash(shard_key) % N nodes. Uniform distribution, no natural ordering.
PROS
Even load distribution
No hotspots
Simple routing logic
CONS
Range queries need scatter-gather
Resharding costly
No locality
Range sharding
Partition by value range — A–M on shard 1, N–Z on shard 2, or by date.
PROS
Range queries fast
Simple mental model
Good for time-series reads
CONS
Hotspot on latest shard
Uneven load
Manual rebalancing
Consistent hashing
Nodes placed on a ring. Each key maps to nearest node clockwise. Virtual nodes even the load.
PROS
Minimal key remapping
Add/remove nodes gracefully
Used by Cassandra, DynamoDB
CONS
Uneven without vnodes
Complex to implement
Hot node risk
Directory / lookup sharding
A lookup service maps each key to its shard explicitly. Full control over placement.
PROS
Full flexibility
Easy custom rebalancing
Supports heterogeneous nodes
CONS
Lookup table = SPOF
Extra hop per request
Cache the lookup or pay latency

Shard key selection

Good shard keys
High cardinality · Even distribution · No natural hotspot
user_id order_id UUID device_id
Bad shard keys
Low cardinality or monotonic = hotspot guaranteed
status country created_at boolean
Timestamp hotspot fix
All writes land on the latest shard. Solutions:
hash prefix + timestamp bucket by time + hash write spreading

When to shard vs replicate

Replicate when
Read traffic is high, writes are manageable
Data fits on a single node (<1TB)
Single-node write QPS is not exceeded
You want simplicity first
Shard when
Write throughput exceeds single node limit
Data volume > 1TB or growing fast
Single-node read replicas are not enough
You need horizontal write scale

Side-by-side analysis

Cache vs replication vs combination

Cache only
Read-heavy workload, data is mostly static or tolerates stale reads
Use when
Hit rate > 80%
Read:write ratio > 20:1
Latency SLA < 5ms
e.g. product catalog, session store
Replication only
Reads distributed across replicas, writes go to primary
Use when
Read:write ratio 5:1–20:1
Strong consistency needed
Data too dynamic to cache
e.g. financial ledger, inventory
Cache + replication
Cache absorbs read spikes, replicas handle cache misses and consistency
Use when
Mixed read/write workload
Cache hit rate 60–80%
Scale beyond single node
e.g. social feed, user profiles
